## 核心原则

### 0. 先独立思考，再参考外部评价

当用户提供外部评价（DeepSeek网页评价、AI网页检测报告等）让我参考时：
- **必须先独立分析**，形成自己的结论和评分
- **然后再看外部评价**，对比差异
- **最后给出综合判断**，标注哪些观点来自自己、哪些与外部一致/不一致

**严禁行为：跳过独立思考直接引用外部评价的结论。**
外部评价是参考，不是标准答案。如果我自己的分析与外部评价不同，说明分歧点并给出理由，
而不是简单地"外部说X所以X是对的"。

**2026-04-30 案例：** 评估 report-10 时，直接引用了 DeepSeek 的 94 分评分框架，
没有先给出自己的独立量化分析。用户指出这是"多次犯的错误"。
**教训**：不管外部评价是谁写的、看起来多么权威，必须先自己思考再参考他人。

### 1. 每一个概率数字必须有可追溯的来源

系统中任何概率数字（CPT值、权重、P(high)等）的合法来源只有三种：
- `cuad_empirical` — 从 CUAD 数据集的 510 份合同中统计得出
- `contractnli_empirical` — 从 ContractNLI 数据集中统计得出
- `expert_estimated` — 专家人工估计（标注清楚，可被后续数据替换）

**严禁行为：为了让报告"看起来更好"而手动调整数字。**
这是数据造假，和被批评的"LLM₂编造反事实数字"是同一性质的错误。

正确做法：
- 如果某个数值不合理 → 从数据集中找统计依据来校准（如 P2.1 的 CUAD 共现统计）
- 如果暂时没有数据 → 保留 expert_estimated 标注，等有数据再改
- 如果呈现效果不好 → 改进呈现方式，而不是改进数字本身

### 2. 提方案前先查证据

在提出"应该把X改成Y"之前，先检查：
- 这个改动有数据依据吗？
- 有没有外部评价（如 deepseek网页评价）支持或反对这个方向？
- 之前有没有人尝试过并得出结论？

**2026-04-29 案例：** 提出"手动调大 Noisy-OR 权重"和"恢复致命等级"
- **错误**：没有数据依据就建议调权重；没查 deepseek 评价就建议恢复致命等级
- **根因**：急于给出方案，跳过了"这个改动有依据吗"的自查
- **教训**：方案中的每一项都要问自己"凭什么"——凭数据？凭外部评价？凭专家经验？如果都凭不上，就不要提

### 3. 区分"真问题"和"呈现问题"

- **真问题**：数据源有误、模型结构有缺陷、计算逻辑有bug
- **呈现问题**：数字不好看、标签不够有力、格式不够清晰

真问题需要通过数据或代码修复。呈现问题可以通过调整prompt、改进展示方式解决。
**把呈现问题当成真问题去修，会导致过度工程甚至数据造假。**

**2026-04-29 案例：** 整体基线概率压缩到 20.4%
- 这是 Noisy-OR 聚合的数学特性，不是 bug
- 真问题的解法：P2.1 用 CUAD 共现数据校准权重
- 呈现问题的解法（已完成）：维度级数据在报告中优先展示

### 4. 项目中的数字溯源链

```
数据集 (CUAD/ContractNLI) → CPT 校准器 → bayesian_network_v2.json
                                    ↓
                            pgmpy BN 推理 → 后验概率
                                    ↓
                            run_sensitivity_analysis → 反事实 delta
                                    ↓
                            LLM₂ prompt → 最终报告中的数字
```

链上任何一环的数字修改都必须有依据。特别是 LLM₂ prompt 中已有多条"严禁编造数字"的约束——我们自己修改模型参数时也必须遵守同样的标准。

## 项目约定

### 文件路径
- 项目根目录：`E:\myProgram\BN-Contract-Risk-Analysis`
- BN 配置：`config/bayesian_network_v2.json`
- 工作清单：`worklist/WORKLIST.md`
- 进度记录：`worklist/PROGRESS.md`（每次改动后更新）
- Python 解释器：`.venv/Scripts/python.exe`

### 严禁使用 git worktree
- **所有开发直接在项目根目录进行，不得创建 git worktree 隔离环境。**
- 原因：本项目前后端都直接运行在项目路径下，worktree 中的修改不会自动同步到实际运行环境，导致"改了但没生效"。
- 如果系统自动进入了 worktree 目录，立即 `exit` 退出并在项目根目录重新开始。
- `.claude/worktrees/` 目录已删除，不可重新创建。
- **后端入口：`backend/main.py`（端口 9527）**——不是 `src/contract_risk_analysis/web/server.py`
- 前端配置：`frontend/vite.config.ts`（API 代理到 localhost:9527）
- 启动命令：`.venv/Scripts/python.exe -m uvicorn backend.main:app --port 9527 --reload`

### 测试
- 运行核心测试：`.venv/Scripts/python.exe -m pytest tests/ -q`

### 报告版本管理
- 报告命名：`contract-review-{合同类型}-{序号}.md`
- 存放在 `合同检测报告/` 目录
- 每次生成新报告后与上一版本对比，确认改动方向正确

## 犯错记录

### 2026-04-30：在错误的后端文件上开发，导致 404

**上下文**：P3-P6 阶段新增了大量 API 端点（导出、历史、反馈、双视角等），全部写在了
`src/contract_risk_analysis/web/server.py`。但项目实际运行的是 `backend/main.py`（端口 9527），
两个文件各自创建独立的 FastAPI app，新端点在实际服务上根本不存在。

**为什么错**：
1. 没有先确认项目实际运行的是哪个文件，凭"看到 server.py 有 FastAPI app"就假设它是入口
2. 前端 `vite.config.ts` 代理到 `localhost:9527`，启动脚本 `backend/main.py` 写在 `启动.txt` 里——都没注意到
3. 擅自把端口从 9527 改成 8000，没查项目约定

**正确做法**：
1. 动手改 API 前，先看 `启动.txt` + `vite.config.ts` 确认后端入口和端口
2. 新 API 端点应该加到 `backend/main.py`，不是另起炉灶
3. 端口跟随项目约定（9527），不擅自改

### 2026-04-29：提出无数据依据的手动调参方案

**上下文**：分析 F 报告时发现 overall_contract_risk 的 P(high)=20.4%，维度级数据（35-78%）远高于整体数字。

**错误行为**：提出 P0 优先级"调整整体基线概率"，实质是建议手动调大 Noisy-OR 权重让数字更好看。

**为什么错**：
1. Noisy-OR 权重是 expert_estimated，手动调没有数据依据
2. 为了让报告"好看"而改数字 = 数据造假
3. 维度级数据已经解决了呈现问题，不需要造数字

**正确做法**：
1. 先判断是真问题还是呈现问题 → 是呈现问题（Noisy-OR 数学特性）
2. 呈现问题已通过维度级数据解决
3. 如果确实需要改善权重 → 走 P2.1 数据驱动校准路径

### 2026-04-29：未查证据就建议"恢复致命等级"

**上下文**：发现 F 报告从原来的 4 级（致命/高/中/低）变为 3 级（高/中/低）。

**错误行为**：提出 P1 优先级"恢复致命等级"，假定这是退步。

**为什么错**：
1. 没有先查 deepseek 评价——评价明确认为 3 级体系"避免了各报告在致命标准上的不一致"
2. 凭直觉认为"标签多=更好"，忽略了内部一致性的价值

**正确做法**：
1. 先查外部评价再下判断
2. 区分"我觉得不对"和"有证据表明不对"
3. 报告质量的衡量标准是逻辑自洽性，不是标签数量
