## P0：BN反事实产出稳定性 — 消灭波动，消灭"BN无数据"

**问题**：同一份瓷砖购销合同，G报告产出8项反事实，H报告产出0项。当前report-8产出3项但最致命的"责任上限"标注为"BN无数据"。反事实产出不稳定是项目当前最大的技术债务——引擎不稳，上层建筑再漂亮也没用。

**目标**：每份报告中≥5个风险项有BN反事实数据，消灭"BN无数据"标注。

**具体操作**：
1. 排查 `pgmpy_adapter.py` 中反事实模拟触发条件——为什么有时产出8项有时0项？
2. 排查 `bayesian_network_v2.json` 中责任上限、违约金对等性等节点的CPT完整性
3. 对缺少数据源的节点，标注 `expert_estimated` 但确保BN推理引擎能产出反事实delta
4. 如现有BN结构不支持某维度反事实，评估新增中间节点或调整图结构
5. 验证：用同一份瓷砖购销合同跑管线，反事实项数从3→≥5

**完成标准**：LLM₂报告中不再出现"BN无法对此维度进行模拟"。

## P1：反事实推导链可视化 — 解决"伪量化"信任危机

**问题**：report-8写"条款失衡风险从52.3%降至8.9%，降幅43.3%"，但读者不知道为什么。DeepSeek评价曾批评这是"数字游戏"。不是数字不对——是读者看不到推导过程所以不信任。

**目标**：每个反事实数字附带可追溯的推导链，让读者理解"这个数字是怎么来的"。

**具体操作**：
1. 在每个反事实项后增加推导链脚注：`证据状态 → CPT先验 → 推理引擎 → 后验delta`
2. 形式示例："（推导：当前条款状态=缺失 + CUAD统计中该条款缺失时82%合同出现纠纷 → 改善后状态=完整约定 → pgmpy推理delta = -43.3%）"
3. 在报告附录中增加"反事实推导链汇总表"
4. 正文保留精确百分比（不改星标），推导链解决信任问题

**完成标准**：非技术读者看到推导链后能理解"这个数字有依据"，不再感觉是黑箱输出。

## P2：报告质量门禁 — 消灭"哑火报告"

**问题**：H报告因为BN哑火（0项反事实），方法论退步到传统模式。如果用户花钱买了系统，某天生成的报告突然没有量化数据，他会觉得系统坏了。

**目标**：管线中增加质量门禁，确保每份报告达标才能输出。

**具体操作**：
1. 在管线 `web/server.py` 的 `/api/v2/review` 中加入反事实产出检查
2. 如果 BN 产出 < 3 项反事实，触发告警日志
3. 设置 fallback 策略：至少确保5个核心维度（财务暴露、履约交付、法律可执行性、争议处置、条款失衡）都有反事实
4. 报告中 BN 未覆盖的维度明确标注原因（节点缺失？CPT缺失？推理失败？）

**完成标准**：每份通过管线的报告反事实项数 ≥ 3，且核心5维度100%覆盖。

## v2.6：报告法律质量提升 + 战略层能力

> 基于 report-10（89→94分）的独立评估发现：Prompt 精简使法律分析质量大幅回升，
> 但存在三个扣分点和一个更高维度的缺失：激进建议缺乏让步梯度、BN 反事实覆盖不足、
> 具体数字规则缺乏法律推理依据、缺少战略级博弈思维。

### P0：数字规则从"硬编码"改为"法律推理规则"

**问题**：当前 prompt 中部分规则嵌入了具体数字（如"违约金降至 5%"、"异议期 7 天"），
这些数字不是法律规定的，是拍脑袋的商业判断。混淆了"法律原则"和"商业判断"。

**修复**：
- 审计 LLM₁/LLM₂ prompt 中所有具体数字
- 凡是非法律明文规定的数字 → 改为推理规则
- 法律有明文规定的（如第 622 条"过短检验期仅视为外观异议期"）→ 保留
- 法律无明文规定的 → LLM 结合标的物属性、行业惯例自行推理

**完成标准**：prompt 中不再出现没有法律依据的具体数字。

### P1：让步梯度规则

**问题**：report-10 建议"违约金清零"——技术上完美封堵了风险，但对方绝不可能接受，
反而可能导致整份修改方案被拒绝，连核心保护（责任上限）也一起落空。

**修复**：在 prompt 中增加一条规则：
> 当建议削弱或消除对方的某项合同权利时，给出两个版本：
> - 开盘立场：我方最理想的目标
> - 可接受底线：仍保护我方核心利益的最低标准
> 例如：违约金从 10%→0% 是开盘立场，但必须标注"若对方坚决不接受，可退让至 5%"

**完成标准**：修改建议不再出现"一刀切"式的极端立场。

### P2：战略层推理框架（可选开关）

**问题**：当前报告能识别风险 + 建议修改，但缺少战略级思维：
- 哪些优势的边际价值已递减，可以策略性让步？
- 哪些风险是不对称致命的，必须清零？
- 如何用让步换保护？

**修复**：在 LLM₂ prompt 中增加可选的战略推理框架：
```
战略评估（三个问题）：
1. 筹码识别：哪些条款优势的"边际价值"已递减？
2. 风险优先级：哪些风险是"不对称致命"的？
3. 交换方案：用(1)换(2)，形成具体交易建议。
```
- 默认关闭（法务用户只需要技术分析）
- 通过 `strategy_mode=true` 请求参数开启
- 框架是合同类型无关的——LLM 每份合同从头推理

**完成标准**：开启策略模式后，报告新增"谈判筹码与策略建议"章节。

### P3：BN 反事实覆盖诊断 + 质量门禁升级

**问题**：report-10 只有 2 项反事实（report-7 有 5 项）。需要确定是卖方视角天然少
还是 BN 引擎有遗漏。且质量门禁只看总数不看"致命风险是否被覆盖"。

**修复**：
- 诊断：同合同买方/卖方各跑一次，对比反事实数量，确定根因
- 质量门禁升级：不仅检查总数≥3，还检查"手动评级最高的 3 个风险项中至少 2 项有 BN 反事实"
- 卖方报告自动在"BN 数据说明"中解释"卖方视角反事实天然较少"

**完成标准**：每次管线运行后，致命风险项的 BN 覆盖率 ≥ 66%。

### P4：企业红线推理框架

**问题**：当前没有"企业红线"概念——某些条款在法律上允许但在商业上对客户是毁灭性的。

**修复**：在 LLM₂ prompt 中增加企业红线推理维度：
> 对每个风险项，评估：
> 1. 可量化损失：最坏情况下损失是否超过合同金额的 X 倍？
> 2. 不可逆性：能否通过后续补救挽回？
> 3. 触发概率：是否在正常履行中就有较大概率被触发？
> 
> 同时满足"损失巨大 + 不可逆 + 高触发"的 → 标注为"🔴 企业红线，不可妥协"。

**完成标准**：每份报告中至少 1 条底线条款附带"企业红线"推理说明。

### P5：民法典合同编精选参考文件

且用户无法验证 LLM 的法律引用是否准确。不需要 1260 条全文，需要合同相关的精选集。

**修复**：
- 创建 `config/civil_code_reference.md`——合同审查常用的 30-50 条精选
- 包含：合同订立、效力、履行、违约责任、买卖合同、争议解决
- 在 LLM₂ prompt 中指明此文件路径，作为法律校验源
- 每条引用应附带条文号 + 简要解读而非原文全文

**完成标准**：报告中每条法律引用均可在精选集中查证。

### P6：联合概率使用指导优化

## v2.8：合同类型分层路由 + 公司红线配置化

> 基于对当前项目与销售合同最新报告（report-11）的综合评估，确认“合同类型节点路由”值得做，
> 但正确方向不是固定 20 节点硬裁剪，而是“通用核心节点 + 合同类型节点包 + 正文触发补集 + 低置信度回退”。
> 同时，“公司红线”应从 prompt 内联规则升级为独立配置层，并区分硬红线与推理指引。

### P0：合同类型分层路由最小闭环

**问题**：当前 `ai_review.py` 的 `_build_bn_checklist()` 会把全部 evidence-layer 节点统一展开，
销售合同也会看到煤炭、采购、软件许可等节点，LLM₁ 噪音偏高，容易出现“泛化补条款”。

**修复**：
- 新建 `config/contract_type_routing.yaml`
- 设计三层节点集：`universal_core` / `contract_types` / `text_triggers`
- 实现合同类型候选识别：标题/首段 + 正文关键词
- `_build_bn_checklist()` 改为支持“核心节点 + 类型节点包”输出

**完成标准**：销售/采购/煤炭三类合同均能优先看到各自高相关节点，但通用核心节点不会丢失。

---

### P1：正文触发补集机制

**问题**：仅靠主合同类型路由，容易漏掉混合型合同中的特殊条款。
例如销售合同正文中出现“热值”“收到基”，若仍只走销售节点包，会漏掉煤炭专属节点。

**修复**：
- 在 routing 配置中新增 `text_triggers`
- 识别“热值/收到基/保密信息/技术资料/调价公式”等强触发词
- 命中后把对应节点补回审查清单

**完成标准**：混合型合同不会因主类型判断正确而漏掉特殊高价值节点。

---

### P2：低置信度回退机制

**问题**：合同标题与正文不一致、命中多个类型、节点过少时，硬裁剪会直接造成漏检。

**修复**：
- 设定低置信度阈值（如 `< 0.6`）
- 多类型并列时取并集
- 当类型节点包过 sparse 时，回退到 `expanded_then_full`
- 在 debug / quality_gate 中记录 fallback 原因

**完成标准**：路由的目标是降噪，不是删掉世界；低置信度场景下宁可多看，不可漏看。

### P3：公司红线配置化（硬红线 + 推理指引）

**问题**：当前“企业红线”更多是 prompt 内联规则，还没有独立配置层。
同时，若把责任上限、异议期、违约金等全部写死为具体数字，会混淆法律原则与商业判断。

**修复**：
- 新建 `config/company_redlines.yaml`
- 每个合同类型下拆为：
  - `hard_rules`：禁止无限责任、间接损失必须排除、质保金不能完全依赖主观验收等
  - `reasoning_hints`：责任上限比例、异议期长度、调价机制是否需要等，由 LLM 结合行业与标的推理
- 在 LLM₁ / LLM₂ 中分别注入：前者用于识别，后者用于定级与谈判表达

**完成标准**：报告能区分“企业不可妥协底线”和“一般优化建议”，且不再大量硬编码具体数字。

## v2.7：LLM₁ 审查覆盖强制化 — 消除风险遗漏

> 基于 report-10(94分) vs report-11(88分) 对比发现：核心差距不在 prompt 规则，
> 而在 LLM₁ 有时静默跳过关键维度（质保期缺失、伪有利条款、价格调整机制）。
> 同一合同、同一立场，该覆盖的风险维度应该每次一样。

### 核心设计：沉默即违规

**放弃 universal/conditional 标签。** 没有真正的"通用型"节点——质保期对 NDA 是噪音，
风险转移对劳动合同是噪音。用标签预判重要性不如让 LLM₁ 读合同后自己判断。

唯一的规则：**57 个检查清单项，每一项都必须被回应。分析 或 N/A+理由，不允许静默跳过。**

| 回应方式 | 含义 | 示例 |
|------|------|------|
| 分析 | 该节点适用于本合同，纳入风险分析 | "质保期缺失 → 买方可在合理期限内随时索赔" |
| N/A + 理由 | 该节点不适用于本合同，主动说明原因 | "源代码托管：N/A，本合同为瓷砖购销，不涉及软件交付" |
| ~~静默跳过~~ | ~~违规~~ | 不允许 |

### P0：LLM₁ 检查清单强制化

- `_build_bn_checklist()` 输出格式：每一项前面加 `[ ]`，要求逐一回应
- Prompt 规则：**"对以下每一项，你必须给出明确结论（分析该风险 / N/A+简短理由）。不得跳过任何一项。跳过 = 审查不完整。"**
- 加两行盲区提示（不膨胀 prompt）：
  1. "卖方必须关注质保期。无质保期 = 买方可在'合理期限'内随时索赔。"
  2. "任何表面有利的条款追问法律效力。法院不支持 = 伪有利，标注为风险。"

### P1：管线关键词冲突检测

- LLM₁ 输出后检查：如果合同文本中出现某节点相关的关键词（如"质保"），但 LLM₁ 对该节点标注了 N/A → 告警
- 关键词映射从 BN 节点 label + CLAUSE_TYPE_HINTS 自动生成

### P2：管线覆盖验证

- `_run_v2_pipeline` 检查 57 个节点中被回应的比例（分析 or N/A）
- 回应率 < 80% → quality_gate WARNING
- 回应率 < 60% → quality_gate ERROR，标注"审查覆盖严重不足"

### 执行顺序

```
P0（检查清单强制化+盲区提示） → P1（关键词冲突检测） → P2（管线覆盖验证）
```
