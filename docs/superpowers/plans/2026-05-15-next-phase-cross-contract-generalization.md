# 后续工作方案：买卖合同基线收口 + 跨合同类型泛化验证 + 中长期方向

> 生成日期：2026-05-15
> 基线版本：v2.14（Release v2.14）
> 方案依据：六份外部评测（含四份更新）+ 两轮独立评测 + 项目 CLAUDE.md 原则

---

## 一、背景

### 1.1 当前状态

买卖合同审查报告已演进至 v22（v2.14 基线）。五份独立评测（豆包 95.85 / 腾讯元宝 92.5 / MiniMax 9.05 / Gemini 9.30 / 独立评测 87.30）一致将 v22 列为最优。v2.13→v2.14 完成了 P0-P2 共 9 项质量提升，BN 术语去学术化、签署底线数字纪律、攻击预判去模板化、合同类型感知、缺失条款分层、立场稳定性回归、Redline 对照表、评测 prompt 标准化均已落地。

### 1.2 核心判断

**买卖合同优化已接近架构上限。** 从 v1（含致命立场错误）到 v22（制度化的工业级产品），本轮会话完成了"代码层硬编码消除 → prompt 层纪律固化 → 规则层护栏补强 → 评测体系标准化"的完整闭环。继续在买卖合同上磨分数的边际收益极低。

**下一阶段的真正竞争维度是"同一套架构在不同合同类型上能否稳定产出 80+ 分报告"。**

## 二、三层优化路线

### 第一层：当前架构内收口（投入低、边际收益中等）

三项目标是在不改变架构的前提下，消除最后几个已知瑕疵。

#### 1-1：第五章退让阶梯进一步方向化

**现状**：v22 的 5.3.2 节退让阶梯仍包含"降至 20%""降至 40%""保留 80%"等具体百分比，读起来仍有固定模板感。

**修改方案**：
- 将退让阶梯中的目标百分比改为引用合同原文锚点 + 方向性表述
  - "降至 20%"→"降至覆盖当前资金敞口的更低水平（当前合同预付款为 80%）"
  - "降至 40%"→"降至有履约保函覆盖的中间水平"
- 在 `_build_combined_prompt()` 退让阶梯指令中增加："退让阶梯中的目标数字必须写为'从当前合同的 X% 降至...'（X 来自合同原文），目标值写为方向+保护条件而非具体百分比"

**涉及文件**：`report_writer.py`（prompt 约束）

**验证**：重跑 v23，检查 5.3.2 节是否不再出现独立具体百分比

---

#### 1-2：第四章防守筹码数据合并去重

**现状**：v22 第四章呈现了 8 项 BN 改善预估，其中 4 条（责任上限强度 / 损害赔偿暴露 / 管辖安排 / liability_cap）核心结论完全相同——"防守筹码，严禁主动修改"。信息冗余度约 50%。

**修改方案**：
- 在 `_build_combined_prompt()` 第四章指令中增加去重约束：
  "如果多条 BN 改善预估的核心结论相同（如'严禁主动修改'），将它们合并为一个段落，格式：'以下 N 项数据均指向同一结论：XXX'"
- 不改变数据完整性，只改变呈现方式

**涉及文件**：`report_writer.py`（prompt 约束）

**验证**：重跑 v23，检查第四章是否为"结论相同"的多条数据提供了合并段落

---

#### 1-3：Redline 附录接入主报告输出

**现状**：`_build_redline_appendix()` 函数已在 P2-2 实现，但 `generate_combined_report()` 未调用。Redline 对照表仅存在于测试中。

**修改方案**：
- 在 `generate_combined_report()` 的 `generate_multi_format_reports()` 调用中，将 redline appendix 作为可选产出
- 在 `backend/routers/review.py` 的 API 响应 `fact_sheet` 中增加 `redline_appendix` 字段
- 前端可后续选择展示

**涉及文件**：`report_writer.py`、`backend/routers/review.py`

**验证**：端到端调用 API，确认响应中包含 `redline_appendix` 字段且内容非空

---

### 第二层：跨合同类型泛化验证（投入中等、边际收益最高）

这是当前项目的真正瓶颈。v22 在买卖合同上评分高，但在其他合同类型上的表现完全未知。

#### 2-1：选取 3 份不同合同类型的真实样本

| 类型 | 标的物属性 | 验证重点 |
|------|-----------|---------|
| 服务外包合同 | 轻资产服务（P1-1 light_service） | 资产类型路由是否正确？付款建议是否合理（不应有"降低预付款至 30%"）？ |
| 技术开发合同 | 定制化开发（P1-1 custom_equipment） | 知识产权归属是否被正确识别为核心条款？立场是否稳定？ |
| 租赁合同 | 标准设备/不动产 | 缺失条款分层是否适配？站姿稳定性是否守住？ |

#### 2-2：逐份验证清单

对每份新合同类型生成的报告，必须通过以下 checklist：

| 检验项 | 方法 |
|--------|------|
| P0-3：第四章无学术术语 | 全文搜索 P(high)/反事实/维度级/BN模拟 |
| P0-1：第六章签署底线方向化 | 全文搜索固定阈值（30%/10%/20%以下 等），确认仅在合同原文引用时出现 |
| P0-2：第五章攻击话术非模板 | 与买卖合同版逐句对比，确认话术不同 |
| P1-1：资产类型路由正确 | 检查 `_detect_asset_type_context` 返回值 |
| P1-3：缺失条款分层 | 确认 P0/P1/P2 三级标注正确且合理 |
| P2-1：立场稳定性 | 确认 liability_cap/jurisdiction/damages_exposure 对买方为 favorable |
| 客户版清洁度 | 无 ISSUE-/DeepSeek/渲染器备注/TODO/TBD |

#### 2-3：根据验证结果调整配置

- 若某合同类型资产路由错误 → 调整 `contract_type_parameters.yaml` 关键词
- 若某合同类型缺失条款分层不合理 → 调整 `missing_clause_priority.yaml`
- 若某合同类型立场反转 → 排查 `adjudicate.py` 的 `_apply_party_aware_rules`
- 若某合同类型 BN 护栏分类错误 → 排查 `BN_INTERPRETATION_RULES`

### 第三层：中长期方向（投入高、边际收益递减前值得布局）

#### 3-1：多条款联动 BN 反事实模拟

**现状**：BN 反事实只模拟单一维度改善（"改善责任上限 → 财务暴露风险降低 X%"），但实际谈判是"降预付款 + 提质保金 + 延风险转移"的组合拳。

**方案**：
- 修改 `consistency_validator.py` 的 `run_counterfactual_simulation`，支持同时设置多个 evidence 节点
- 输出联合反事实的维度级 delta
- 在第四章增加"组合改善效果预估"段落

**价值**：谈判策略从"静态清单"变成"动态沙盘"——客户可以问"如果我让步了 A 和 B，换取了 C 和 D，整体风险怎样变化"。这是腾讯元宝、Gemini、豆包三份评测共同指出的方向。

#### 3-2：行业参数库积累

**现状**：`contract_type_parameters.yaml` 中的参数区间是我基于法律常识估计的，未经真实行业数据校准。

**方案**：
- 积累不同行业的真实合同样本
- 从样本中统计实际预付款比例、质保金比例、质保期、违约金比例的分布
- 用统计分布替代人工估计值

#### 3-3：谈判模拟器原型

**现状**：报告是静态的，生成后不变。

**方案**：
- 构建交互式界面（可以是命令行或简单 Web）：输入"预付款从 80% 降到 X%""质保金从 5% 升到 Y%"，实时计算整体风险概率变化
- 不需要 LLM，只需要改 BN 推理层的 evidence 组合 + 一次推理
- 可作为 API 端点提供

## 三、执行顺序

```
第一层（立即）：
  ├── 1-1 退让阶梯方向化
  ├── 1-2 BN 数据去重
  └── 1-3 Redline 接入
        ↓ 买卖合同基线收口，生成 v23
第二层（下一主线）：
  ├── 2-1 选取 3 份新合同类型样本
  ├── 2-2 逐份跑端到端 + checklist 验证
  └── 2-3 根据发现调整配置
        ↓ 跨合同类型泛化验证完成
第三层（中长期）：
  ├── 3-1 多条款联动 BN
  ├── 3-2 行业参数积累
  └── 3-3 谈判模拟器原型
```
