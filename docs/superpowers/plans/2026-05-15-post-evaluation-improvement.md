# 后续工作方案：基于六方评测的买卖合同审查报告质量提升

> 生成日期：2026-05-15
> 基线版本：v2.13（Release v2.13）
> 方案依据：六份外部AI网页评价 + 独立版本对比评测 + 项目 CLAUDE.md 原则

---

## 一、背景与目标

### 1.1 当前状态

买卖合同审查报告已演进至 v21（v2.13 基线），六份独立外部评价（DeepSeek、豆包、腾讯元宝、MiniMax、Gemini、AI网页端）一致将 v21 列为最优版本。v21 的核心优势在于**五层系统性护栏**（BN 使用层级分类、筹码联动矩阵、Dossier 净化器、客户版清洁度约束、反事实使用纪律），而非单次 LLM 的策略发挥。

### 1.2 方案目标

在 v21 基础上，针对六份评测共同指出的改进方向，完成一系列可落地、可验证的质量提升。**核心原则：不做过度拟合单份合同的优化，不做不可复现的手工修改。**

---

## 二、待执行项

### P0：立即执行（必须 + 急迫 + 做后明显提高）

#### P0-1：签署建议底线数字去硬编码

**现状**：v21 第六章签署建议仍包含绝对数字（"预付款比例未能降低至 30% 以下""质保金比例提高至 10%""压至 20% 以下"），这些数字无合同文本来源，且在不同合同中应有所不同。

**六方评价关联**：Gemini 指出"暴露系统指令痕迹会削弱报告权威性"，豆包指出"签署底线的数字应基于行业参数"。

**修改方案**：
1. 在 `_build_dossier()` 的 `signing_forbidden`/`signing_acceptable` 生成逻辑中，对包含百分比的底线文本应用 `_rewrite_unsourced_payment_thresholds()` 净化
2. 签署底线中的数字应优先来自 Dossier 定量锚点（如合同原文的 80%/5%），否则写"建议人工根据商业背景确定"+"不低于覆盖当前资金敞口的水平"
3. 修改 `_build_combined_prompt()` 中第六章相关指令：签署底线可以写方向和条件，不得写系统无法追溯的绝对百分比

**涉及文件**：
- `src/contract_risk_analysis/review/report_writer.py`：`_build_dossier()` 底线生成逻辑
- `tests/regression/test_judgment_regression.py`：新增 `test_signing_guardrails_do_not_contain_unsourced_thresholds`

**验证标准**：用同一份买卖合同和另一份低预付款合同分别跑端到端，确认两份报告的签署底线数字不同（证明不是模板硬编码）

---

#### P0-2：第五章对手攻击预判去模板化

**现状**：v21 第五章对手攻击预判的部分话术段落（如"管辖地需改为被告所在地"的攻击话术）在 v17/v18/v19/v20 中高度相似，提示 LLM 可能复用了训练数据中的固定话术模板。

**六方评价关联**：腾讯元宝指出"报告-21成功将风险清单升级为谈判作战手册"，但 Gemini 和豆包都提示"需要更多情景化推演和合同特定攻击话术"。

**修改方案**：
1. 在 `_build_combined_prompt()` 第五章策略区增加约束："对手律师话术必须引用当前合同的至少一个具体条款号和至少一个本合同独有的具体数字（如 80%/1,228万/5%/1535万 等），不得使用'显失公平''行业惯例'等无差别攻击模板"
2. 在 `_free_review_prompt()` LLM₁ prompt 中增加约束："counterparty_attack_vector 必须从当前合同的具体条款措辞出发推演，不得套用'三板斧'等固定攻击标签"
3. 扩充 `_run_pre_render_consistency_checks()` 增加 Check 10：检测连续 N 个版本中第五章攻击话术的文本相似度，超过阈值则发出跨版本模板化警告

**涉及文件**：
- `src/contract_risk_analysis/review/ai_review.py`：LLM₁ prompt
- `src/contract_risk_analysis/review/report_writer.py`：LLM₂ prompt 和 pre-render checks
- `tests/review/test_report_writer_negotiation_chip.py`：新增对应断言

**验证标准**：用两份不同行业合同（买卖合同 + 服务合同）分别生成报告，确认两份报告的第五章攻击话术在具体引用条款号、数字和行业背景上有显著差异

---

#### P0-3：第四章 BN 术语去学术化

**现状**：v21 第四章仍包含"P(high)""反事实模拟""维度级 delta""BN 模拟效果"等学术/系统术语，对外交付给非技术客户时降低报告的可读性和权威感。

**六方评价关联**：Gemini 将此列为 P1 立即执行项（"管理层不需要知道算法原理，只需要知道结论和对价"），豆包和腾讯元宝也分别提到了"去学术化"和"商业语言转译"。

**修改方案**：
1. 在 `_build_combined_prompt()` 第四章格式指令中增加约束：
   - "P(high)" → "风险概率"
   - "维度级数据" → "具体风险维度的改善效果"
   - "整体级数据（辅助参考）" → "合同整体的参考效果"
   - "反事实分析" → "关键条款改善效果预估"
   - "delta 降幅" → "预期改善幅度"
2. 同时保留"贝叶斯网络"一词（作为方法论溯源，在附录中解释）
3. 第八章附录中的 BN 术语可以保留，但第四章正文必须使用商业语言

**涉及文件**：
- `src/contract_risk_analysis/review/report_writer.py`：`_build_combined_prompt()` 第四章指令段
- `tests/review/test_report_writer_negotiation_chip.py`：更新术语检测断言

**验证标准**：重跑真实样本，确认第四章正文中不出现"P(high)""反事实""维度级""BN 模拟效果"等学术标记

---

### P1：中期规划（必须 / 有用但非急迫 / 做后有提高但需较多投入）

#### P1-1：合同类型感知的差异化分析

**现状**：当前系统无论审查设备采购还是服务外包，底层风险评估参数和推荐区间是同一套。标准工业设备（如本案水质检测仪）和高度定制化设备的预付款合理区间、质保期基准、违约金容忍度应该有系统性的差异。

**六方评价关联**：豆包指出"BN 模型的 CPT 参数基于通用合同数据集，建议积累行业数据进行校准"，腾讯元宝指出"应建立合同类型路由以差异化分析"。

**修改方案**：
1. 在 `src/contract_risk_analysis/review/ai_review.py` 的 `detect_contract_type_routing()` 中增加"标的物属性"维度（标准件/定制件/大宗商品/轻资产服务）
2. 新增配置文件 `config/contract_type_parameters.yaml`，定义不同标的物类型下的参数区间（如预付款合理上限、质保期基准、违约金合理范围）
3. 在 LLM₁ prompt 和 Dossier 构建中接入标的物属性参数，影响推荐措辞的"合理区间"引用
4. 在 `quantification.py` 中增加标的物属性感知的交换比率计算

**涉及文件**：
- `src/contract_risk_analysis/review/ai_review.py`
- `src/contract_risk_analysis/review/quantification.py`
- `config/contract_type_parameters.yaml`（新建）
- `tests/review/test_quantification.py`

**验证标准**：用买卖合同（标准设备）和一份定制化设备合同分别生成报告，确认两份报告的预付款合理区间引用、质保期建议和违约金判断存在基于标的物属性的合理差异

---

#### P1-2：策略区与 Dossier 数字一致性自动校验

**现状**：v21 第五章策略区中 LLM 自发生成的退让阶梯数字（如"10%→30%"）并非全部来自前四章 Dossier 数据，存在"分析说一套、策略说另一套"的风险。

**六方评价关联**：腾讯元宝指出"BN 数据与策略的联动应形成'数据-洞察-策略'增强闭环"，Gemini 指出"策略区的数字应在分析区有可追溯来源"。

**修改方案**：
1. 扩展 `_run_pre_render_consistency_checks()` 增加 Check 11：提取第五章所有百分比/金额数字，与 `_allowed_numeric_phrases(dossier)` 做交叉校验
2. 对于策略区独有（前四章未出现且 Dossier 未提供）的数字，发出日志警告并在 Dossier 的 `internal_conflicts` 中记录
3. 在 `_build_combined_prompt()` 中增加自检指令："在完成第五章退让阶梯后，逐项自检：每个数字是否已在前四章或 Dossier 中出现过？如果没有，将其替换为方向+保护条件的表述"

**涉及文件**：
- `src/contract_risk_analysis/review/report_writer.py`
- `tests/regression/test_judgment_regression.py`

**验证标准**：重跑真实样本，确认日志中不再出现"策略区独有数字"警告

---

#### P1-3：缺失条款的优先级分层

**现状**：v21 列出了 10 项缺失条款，但没有区分哪些是签署底线、哪些可谈判、哪些可忽略。用户（法务团队）拿到后仍需自己排序。

**六方评价关联**：豆包和腾讯元宝分别提到"整改计划的优先级分层"和"缺失条款的分类管理"。

**修改方案**：
1. 在 `_build_dossier()` 的缺失条款处理逻辑中增加分类规则：
   - P0（签署底线）：知识产权归属、保密义务（如标的物涉及技术秘密）
   - P1（强烈建议）：不可抗力详细条款、适用法律明确、送达地址
   - P2（可谈判）：保险条款、审计权、排他性条款
2. 分类依据存储在 `config/missing_clause_priority.yaml`（可按合同类型覆盖）
3. 在 Dossier 渲染区（`_fmt_dossier_section`）和 LLM₂ prompt 中将缺失条款按优先级展示

**涉及文件**：
- `src/contract_risk_analysis/review/report_writer.py`
- `config/missing_clause_priority.yaml`（新建）
- `tests/regression/test_judgment_regression.py`

**验证标准**：缺失条款在报告中按 P0/P1/P2 分组展示，每组有明确的修改紧迫性说明

---

### P2：低优先级（有益但非必须 / 不紧迫 / 边际收益有限）

#### P2-1：跨版本回归的"立场稳定性"自动检测

**现状**：v1→v21 演进中，某些条款的立场曾经反复（如 v1/v4 将"无责任上限"误判为买方风险，v8+ 纠正），目前无自动检测。

**修改方案**：在 `tests/evaluation/test_golden_score.py` 中增加"立场稳定性"断言——对于 golden case 中的同一份合同、同一立场，所有后续版本对特定条款（管辖、责任上限、违约金）的判断方向必须与 golden case 一致（favorable/unfavorable 的符号不反转）。

---

#### P2-2：Redline 条款对照表附件

**现状**：v21 中各章的修改建议散落在正文中，法务团队需要手动整理成修订对照表。

**六方评价关联**：Gemini 提出"应在文末生成原条款 vs 推荐修改的双栏对比表，直接供法务复制粘贴至 Word 中生成修订版"。

**修改方案**：新增 `_build_redline_appendix(dossier)` 函数，从 Dossier 的 risk_items 中提取 evidence_text（原条款）和 recommendation（修改建议），生成 Markdown 双栏表格作为附录。纯确定性生成，不涉及 LLM。

---

#### P2-3：评价 prompt 标准化

**现状**：当前六份外部评价使用了不同的评分维度和权重体系，导致同一份报告在不同评价中得分差异巨大（如 v12 在 DeepSeek 92 分 vs MiniMax 71 分）。这不全是报告质量的问题，也反映了评价标准本身的不一致。

**修改方案**：整理现有的 6 份评价 prompt + 我的评分模型，提炼出一份标准化的"合同审查报告质量评测 prompt 模板"，存入 `合同检测报告/评价prompt/` 目录。未来新的外部评价都使用这份统一 prompt，提高可比性。

## 三、执行顺序与依赖关系

```
P0-3（BN 术语去学术化）──→ 无依赖，可立即执行
P0-1（签署底线去硬编码）──→ 无依赖，可立即执行
P0-2（攻击预判去模板化）──→ 无依赖，可立即执行
    │
    ├──→ P1-2（策略-分析数字一致性校验）──→ 依赖 P0-1
    ├──→ P1-1（合同类型感知）──────────→ 独立执行
    └──→ P1-3（缺失条款分层）──────────→ 独立执行
            │
            └──→ P2-1 / P2-2 / P2-3 ──→ 可并行执行
```

---

## 四、测试与验收

每项完成后必须通过：
1. 单元/回归测试（已有测试套件不退化）
2. 真实样本端到端重跑（买卖合同 PDF → 确认改动方向正确）
3. 跨合同类型抽样（至少 2 种不同合同类型，确认改动不是买卖合同特化）

## 五、与项目 CLAUDE.md 原则的对齐检查

| CLAUDE.md 原则 | 本方案是否违反 | 说明 |
|---------------|:---:|------|
| 禁止把外部评价当系统答案 | 否 | 六方评价作为参考信息，所有修改方向由我独立判断后确定 |
| 禁止样本硬编码 | 否 | P0-1 正是为了进一步消除残留的硬编码 |
| 禁止编造数字 | 否 | P1-2 增加了策略区数字溯源校验 |
| 客户版必须清洁 | 否 | P0-3 去学术化进一步提升客户版质感 |
| 优先结构性规则，不引入复杂特判 | 否 | P1-1 使用配置文件而非代码内特判 |
