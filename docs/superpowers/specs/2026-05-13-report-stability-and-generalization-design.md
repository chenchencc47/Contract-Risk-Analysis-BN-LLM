# 报告稳定性与泛化能力优化方案

> 日期：2026-05-13  
> 项目：BN-Contract-Risk-Analysis  
> 目标：在不把个案判断硬编码进生产规则的前提下，提升合同审查报告的稳定性、泛化能力和正式成品质量，并单独处理 `contract-review-买卖合同-13.md` 的当前降质问题。

---

## 1. 背景与问题定义

近期对买卖合同多版本报告的对比暴露出三类不同性质的问题，必须分开处理：

1. **运行态不一致问题**  
   同一代码仓库在“热修改未重启”和“重启后重新生成”两种运行态下，实际服务进程加载的代码、prompt 和响应结构可能不同。此次 `-12` 与 `-13` 的差异，首先不能简单归因为“同版随机波动”，因为 `-12` 生成时后端未完全重启，而 `-13` 生成时后端已重启，Golden 分数也只在后者中显现。

2. **同版生成漂移问题**  
   即使运行态完全一致，系统仍然存在一定输出波动，主要来自：
   - LLM₁ 对风险、优势、筹码、优先级的自由裁量；
   - LLM₂ 在正式报告中的非零温生成；
   - prompt 中若干法律角色边界仍允许不同表述方式。

3. **报告成品质量问题**  
   最新报告 `contract-review-买卖合同-13.md` 抓住了核心风险，但出现了前后不一致、事实互撞、无来源数字和客户版被内部标签污染等问题。这是“报告质量控制”问题，不应直接通过对本案写死规则来修。

本方案的核心原则是：

- **个案结论只修个案，不进入生产硬编码。**
- **泛化模式进入 production rules；固定样本留在 golden cases；中间用 golden patterns 连接。**
- **系统优化优先解决结构性问题，而不是为这份买卖合同定制答案。**

---

## 2. 设计边界与约束

### 2.1 三层边界必须保持清晰

#### A. Golden Case：固定样本回归

Golden case 的作用是检查“这份已知样本在系统升级后有没有退步”。它可以包含这份买卖合同的具体特征，例如：
- 80% 预付款；
- 1228 万元金额；
- 北京市辖区范围内交付；
- 初验即视为交付。

这些内容**允许出现在回归样本中**，因为它们的作用是回归，而不是泛化。

#### B. Golden Pattern：从多个样本抽象出的风险模式

Golden pattern 的作用是表达可迁移的结构，例如：
- 高额预付款且无有效担保；
- 付款担保结构倒挂；
- 责任上限按审查立场反向判断；
- 最终验收前风险提前转移。

Pattern 应尽量用：
- 比例阈值；
- 节点顺序；
- 责任主体；
- 审查立场；
- 合同类型；
- 是否有担保；
而不是用某个样本的固定金额和固定数字。

#### C. Production Rules：真实新合同检测规则

Production rule 的职责是对任何新合同做方向判断。它只能吸收已经通过多个样本验证的 pattern，不能直接复制某一份合同的人工结论。

### 2.2 禁止进入生产规则的内容

以下内容可用于修当前报告、写 case 备注或人工谈判建议，但**不得直接硬编码为生产规则**：

1. “本案预付款底线必须是 50% / 60%”之类的具体底线数字；
2. “所有合同风险转移必须到终验”这类跨合同类型的绝对化结论；
3. “甲方住所地法院一定最优”这类不考虑合同场景的绝对化规则；
4. 某一份合同的具体 fallback 责任上限数字；
5. 某一份报告里为了增强说服力而写入的估算数字或谈判话术。

### 2.3 本次优化的成功标准

本方案完成后，应满足以下目标：

1. **运行态可识别**：用户能看出报告是在哪个后端运行态下生成的；
2. **新合同可泛化**：生产规则不依赖这份买卖合同的样本数字；
3. **正式报告更稳定**：同一合同多次生成时，核心结论和签署底线保持一致；
4. **客户版更干净**：无内部编号、占位符、无来源数字和事实互撞；
5. **回归评测不误导**：Golden 分数被明确解释为“样本回归匹配分”，而不是通用质量总分。

## 3. 现状诊断

### 3.1 当前已经做对的部分

1. **Dossier 冻结层已经存在**  
   风险项、BN 数据、有利条款、签署底线已经有较明确的结构化承载层，不再完全依赖 LLM₂ 临场发挥。

2. **Party-aware 规则方向正确**  
   对责任上限、管辖、风险转移、付款结构等关键条款，系统已经开始按 buyer/seller 立场反向处理，而不是机械地“一律定性为风险”。

3. **Golden Case / Golden Pattern / Production Rule 的概念已经被区分出来**  
   这是后续泛化能力建设的基础。

4. **部分确定性输出已经落地**  
   修订清单、BN 附录等内容已能从结构化数据直接生成，为进一步减少随机性提供了路径。

### 3.2 当前主要缺陷

1. **生产检测里仍夹杂样本锚点**  
   例如结构性付款担保倒挂检测中仍使用了 `80%`、`1228万` 一类明显带有样本痕迹的信号，虽然并非完全依赖它们，但这会削弱泛化可信度。

2. **Golden 分数的产品语义不够清楚**  
   当前前端展示容易让用户误以为这是“任何合同都通用的质量总分”，而实际上它更接近已有 golden cases 的最佳匹配回归分。

3. **正式法务详版仍有少量随机漂移**  
   报告虽然已经接近稳定，但关键结论仍有被措辞放松、数字展开方式变化、策略边界漂移的风险。

4. **缺少成品级 lint**  
   当前系统更关注法律方向和 BN 使用层级，但对“同一报告内部是否前后打架、是否残留占位符、是否出现无来源数字、是否把内部标签暴露给客户版”的检查还不够强。

## 4. 推荐方案对比

### 方案 A：继续以 Golden Case 为中心修修补补

做法：继续扩展现有买卖合同样本，对评分标准和检测逻辑做局部修补。

优点：
- 上手快；
- 对当前买卖合同最直接有效。

缺点：
- 极易把个案经验误沉淀成生产逻辑；
- 对其他合同泛化能力弱；
- 用户容易把回归样本分数误当成系统通用能力。

结论：**不推荐作为主方案。**

### 方案 B：大规模重构整条审查链

做法：一次性重写规则层、Dossier 层、生成层、评测层。

优点：
- 理论上最彻底；
- 可以获得一套更统一的体系。

缺点：
- 改动面大；
- 容易打断当前已稳定的能力；
- 验证成本高。

结论：**当前不推荐。**

### 方案 C：分层收口的渐进式稳定化（推荐）

做法：保持现有架构，按“规则层 → 评分层 → 生成层 → 成品 lint 层”的顺序，逐步把样本锚点剥离、把成品质量控制补齐。

优点：
- 改动范围可控；
- 不破坏现有已有效的 Dossier / Party-aware / BN guardrail；
- 既能处理当前 `-13` 问题，又能提升新合同泛化能力。

结论：**推荐采用。**

## 5. 通用优化方案

### 5.1 规则层：去样本锚点，增强结构泛化

#### 5.1.1 目标

确保生产规则识别的是“模式”，而不是“这份买卖合同的影子”。

#### 5.1.2 优化内容

1. **去除生产检测中的样本数字锚点**  
   将类似 `80%`、`1228万` 这类属于当前 golden case 的痕迹，从生产检测逻辑中剥离。生产逻辑应优先依赖：
   - 预付款比例阈值；
   - 付款是否发生在实质交付前；
   - 担保是否在付款前存在；
   - 质保金/保证金节点是否后置；
   - 风险转移是否早于与合同性质匹配的实质验收节点。

2. **让 pattern 条件真正进入结构判断**  
   优先用结构字段判断：
   - `prepayment_ratio_gte`
   - `pre_delivery_payment_ratio_gte`
   - `security_before_payment`
   - `payment_before_substantial_delivery`
   - `liable_party`
   - `review_stance`
   - `risk_transfer_before_final_acceptance`

3. **继续将高价值判断从 LLM₁ 裁量迁到规则层**  
   优先收紧以下模式：
   - 高额预付款/付款前置；
   - 付款担保结构倒挂；
   - 责任上限按立场判断；
   - 管辖按立场判断；
   - 到货/外观签收与内在质量责任的冲突。

#### 5.1.3 成功标准

- 新合同即使没有“80%/1228万”这类数字，也能正确识别“高额预付款无担保”风险；
- 同一模式在买方/卖方视角下仍能给出不同方向，而不是回到中性关键词匹配。

---

### 5.2 评分层：重新定义 Golden 分数的产品语义

#### 5.2.1 目标

避免用户把当前 Golden 分数误解成任何合同都通用的“客观总分”。

#### 5.2.2 优化内容

1. **将 Golden 分数明确标注为“回归样本匹配分”**  
   它的含义是：这份报告与当前已有 golden cases 中最相近样本的对齐程度，而不是通用质量总分。

2. **为新合同增加“生产规则命中视图”**  
   与其只显示一个分数，更应展示：
   - 命中了哪些 pattern；
   - 哪些 must_not 没被触犯；
   - 哪些优势条款被正确识别；
   - 哪些结论仍需人工复核。

3. **把回归评分和生产评分拆成两个概念**  
   - 回归评分：验证系统是否在旧样本上退步；
   - 生产命中：说明系统对当前新合同识别到了哪些泛化模式。

#### 5.2.3 成功标准

- 用户知道 Golden 分数是“回归能力指示器”，不是“全局质量真理值”；
- 新合同不会因为碰巧撞上某个 case 的关键词而得到误导性高分或低分。

---

### 5.3 生成层：降低正式报告随机性，分离客户版与内部版

#### 5.3.1 目标

提高同版生成的一致性，并让不同受众看到不同粒度的内容。

#### 5.3.2 优化内容

1. **正式法务详版继续向确定性收紧**  
   对正式客户版报告，应尽量减少不必要的措辞漂移。重点不是让文风更花，而是让：
   - 风险结论稳定；
   - 底线一致；
   - 签署建议不摇摆。

2. **客户版与内部作战版彻底分流**  
   - **客户版**：法务立场明确、结构清晰、语言克制，不出现内部编号、开发标签和过重工程痕迹；
   - **内部作战版**：保留响应筹码、交换矩阵、对手话术、退让阶梯等高密度谈判信息。

3. **把更多可确定部分从 LLM 渲染中抽出**  
   优先确定性化：
   - 风险总览表；
   - 签署禁止条件；
   - 可签署条件；
   - 修订清单；
   - BN 附录；
   - 内部一致性警告。

#### 5.3.3 成功标准

- 同一合同多次生成时，签署底线、核心风险和关键建议稳定；
- 客户版报告不再混入工程内部标记；
- 内部版则允许更强的策略颗粒度。

---

### 5.4 成品一致性校验层：新增报告 lint

#### 5.4.1 目标

在返回给用户前，用确定性规则抓掉“成品味不足”的问题。

#### 5.4.2 优化内容

新增以下通用 lint：

1. **结论一致性检查**  
   同一风险项的开盘目标、可接受中间价、底线值不得前后冲突。

2. **立场一致性检查**  
   前文被明确识别为 `must_fix` 的条款，后文不得被无解释地放松为可接受；前文认定为有利条款的内容，后文不得又写成“该条款未明确”。

3. **占位符检查**  
   禁止正式报告中出现 `【X】`、`TODO`、`TBD` 等占位符。

4. **无来源数字检查**  
   正式报告中的概率、乘数、降幅、估算成本、估算回收率等数字，必须能追溯到 Dossier、BN 输出、数据源或明确的规则来源。没有来源的数字不允许进入正式版。

5. **客户版清洁度检查**  
   面向客户的最终报告不得暴露：
   - `ISSUE-xxxx` 内部编号；
   - canonical type；
   - 调试性说明；
   - 开发标签。

#### 5.4.3 成功标准

- 报告在提交给用户前，先通过成品级 lint；
- 当前 `-13` 暴露出的主要问题，有至少 80% 能被 lint 自动拦下。

---

### 5.5 运行态可观测性：区分“同代码仓库”与“同运行态”

#### 5.5.1 目标

让用户和开发者能区分：“这是模型随机波动”，还是“服务进程实际加载状态不同”。

#### 5.5.2 优化内容

在报告响应和内部元数据中记录：
- 后端启动时间；
- 当前应用版本号或提交标识；
- prompt 版本；
- 是否启用 Golden scoring；
- 输出格式类型；
- 生成时间。

#### 5.5.3 成功标准

- 当用户说“我没改代码，但结果不一样”时，可以先核对是不是同一运行态；
- 避免把热重载、未重启、模块未刷新误判为“模型质量退步”。

## 6. 实施顺序

### 6.1 立即执行

1. **从生产规则检测中去除样本数字锚点**  
   收益：直接降低“硬编码买卖合同样本”的质疑。  
   风险：若替代结构判断不完整，可能短期影响部分命中率，因此需要配套回归测试。

2. **增加成品 lint：占位符、无来源数字、内部编号、事实互撞**  
   收益：最快修复当前 `-13` 这类成品质量问题。  
   风险：过严的 lint 初期可能拦截较多报告，需要逐步校准。

3. **明确 Golden 分数产品语义**  
   收益：避免用户误读当前评分能力边界。  
   风险：无实质风险。

4. **记录运行态元数据**  
   收益：区分“真随机”与“运行态不一致”。  
   风险：无实质风险。

### 6.2 中期规划

1. **将更多高价值判断迁移到结构化 pattern / rule 层**  
   收益：减少 LLM₁ 自由裁量导致的同版漂移。  
   风险：若抽象过度，可能丢失特殊场景灵活性。

2. **拆分客户版与内部作战版**  
   收益：同时提升正式报告可交付性与内部谈判实战性。  
   风险：需要重新梳理输出接口与前端呈现。

3. **建立双跑一致性回归**  
   收益：把“分高但不稳”的问题纳入评测。  
   风险：测试成本增加。

### 6.3 低优先级

1. **进一步优化文风、攻击性和桥接性话术**  
   收益：提升高级法务风格。  
   风险：容易分散注意力，不应先于结构稳定性建设。

2. **引入更强的 LLM-as-judge 评测**  
   收益：对复杂表达质量有更细腻的判断。  
   风险：若过早依赖 judge，反而会掩盖基础规则评测还没扎实的问题。

---

## 7. 验证方案

### 7.1 回归验证

对现有 golden cases 验证：
- must_find 是否仍稳定命中；
- must_not 是否仍不被触犯；
- should_find_advantages 是否不倒退。

### 7.2 泛化验证

对非买卖合同样本验证：
- 生产规则是否仍能识别高额前置付款、责任上限方向、风险转移过早等结构性模式；
- 是否不依赖 80%、1228 万这类样本锚点。

### 7.3 成品验证

对正式客户版报告验证：
- 无内部编号；
- 无占位符；
- 无无来源数字；
- 关键底线前后一致；
- 不出现事实互撞。

## 8. 附录：`contract-review-买卖合同-13.md` 个案修复清单（不产品化）

本附录仅用于修当前最新报告，不得直接沉淀为生产硬编码规则。

### 8.1 需要修复的问题

1. **预付款底线前后冲突**  
   报告内同时出现了 50% 与 60% 两个不同底线口径，必须统一。

2. **风险转移节点被放松**  
   前文将“终验”为核心保护节点，后文又接受了“初验后 30 日或终验取较早者”的放松表达，破坏了前后立场一致性。

3. **管辖条款事实互撞**  
   一处写成“甲方住所地法院已是有利条款”，另一处又写成“争议解决条款未明确，需确认”，必须二选一。

4. **无来源数字进入正式版**  
   包括诉讼成本、回收率、资金成本等估算数字，若无 dossier / BN / 数据来源支持，应删除或明确为假设示例且不进入正式客户版。

5. **客户版被内部编号污染**  
   `ISSUE-xxxx` 等内部标识不应出现在正式客户版报告中。

### 8.2 个案修复原则

1. 只修当前报告表述，不把本案结论上升为通用规则；
2. 所有具体底线值必须保留人工决策空间；
3. 所有修复都应以 Dossier 现有冻结事实为准，不允许为追求好看而新增无来源内容。

## 9. 最终建议

推荐采用“**分层收口的渐进式稳定化**”路线：
