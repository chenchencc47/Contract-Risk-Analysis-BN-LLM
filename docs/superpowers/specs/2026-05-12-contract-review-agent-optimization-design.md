# 合同审查 Agent 系统优化方案

> 日期：2026-05-12  
> 项目：BN-Contract-Risk-Analysis  
> 目标：从“能生成高质量合同审查报告”升级为“事实可核验、立场可裁决、数据可追溯、策略可执行、版本可评测”的合同风控 Agent。

---

## 1. 背景与结论

本项目已经形成了较强的差异化能力：LLM₁ 自由审查、BN 风险推理、Report Dossier、canonicalization/adjudication、NegotiationChip 结构化对象、LLM₂ 受约束报告渲染，以及谈判筹码策略生成。

在买卖合同多版本报告对比中，最新报告已经接近可用的实战型法务报告，但仍暴露出几个系统性问题：

1. 事实层、裁决层、渲染层边界还不够硬；
2. BN 反事实数据容易被误读为主动修改建议；
3. 买方/卖方立场下的同一条款定性仍可能摇摆；
4. 报告输出形态偏“大而全”，管理层、法务、业务谈判所需版本未拆分；
5. 缺少稳定的自动评测集，报告质量提升主要依赖人工对比。

本方案建议将系统升级为六层架构：

```text
合同输入与证据抽取层
→ Dossier 事实清单层
→ 法务裁决层
→ BN 数据与交叉校验层
→ 多形态报告生成层
→ 自动评测与版本对比层
```

## 2. 市面优秀 AI 合同审查 Agent 的可借鉴能力

### 2.1 条款结构化抽取

优秀工具通常先抽取合同类型、审查立场、条款主题、条款编号、原文证据、金额、比例、期限和义务主体。

**可借鉴：** 本项目应继续强化 Dossier，确保每个风险结论都能回到合同原文。  
**应避免：** 只抽取条款但不判断该条款对哪一方有利。

### 2.2 Playbook / 红线规则

成熟工具通常有企业级审查规则，例如预付款比例上限、担保要求、责任上限、管辖地、知识产权、保密条款等。

**可借鉴：** 将规则设计为“合同类型 + 审查立场 + 风险偏好 + 金额规模”的动态规则。  
**应避免：** 把红线规则做成死模板，例如“缺少责任上限 = 一律高风险”。

### 2.3 原文引用与证据链

优秀报告不会只说“存在付款风险”，而会引用条款编号和原文，解释商业后果。

**可借鉴：** 每个核心判断必须绑定条款编号、原文片段、风险解释、修改建议和谈判用途。  
**应避免：** 没有来源地使用“行业惯例”“司法实践通常认为”等确定性表述。

### 2.4 风险、优势、筹码分离

优秀法务报告不只列风险，还会区分：必须修改的风险、必须守住的优势、可交换筹码、响应筹码。

**可借鉴：** 将 negotiation role 固化为结构化字段：`risk`、`advantage`、`bottom_line_chip`、`exchange_chip`、`response_chip`。  
**应避免：** 把所有条款都写成风险，导致谈判立场错误。

### 2.5 多形态输出

成熟工具通常会按使用对象拆分报告：管理层摘要、法务详版、谈判手册、修订条款清单、数据附录。

**可借鉴：** 本项目应从“一份大报告”升级为“同一 Dossier 多版本输出”。  
**应避免：** 所有内容堆进一份报告，导致管理层看不完、业务不会谈、法务难复核。

## 3. 本项目目标架构

### 3.1 合同输入与证据抽取层

目标：保证报告有可靠原文依据。

能力要求：

- 支持 PDF / Word / Markdown / 文本；
- 对扫描 PDF 增加 OCR 预留接口；
- 提取条款编号、页码、原文、主题、金额、比例、期限、义务主体；
- 为每个后续风险项提供 evidence span。

建议结构：

```json
{
  "clause_id": "payment_9_2",
  "clause_no": "第九条第2款",
  "text": "甲方于本合同签订后10日内向乙方支付合同金额的80%作为预付款...",
  "topic": "payment",
  "party_obligation": "甲方付款",
  "amount": 12280000,
  "percentage": 80,
  "evidence_span": {"page": 5, "start": 120, "end": 188}
}
```

### 3.2 Dossier 事实清单层

目标：LLM₂ 写报告前，先获得稳定、结构化的事实底座。

建议补强字段：

```json
{
  "issue_id": "ISSUE-payment-prepay-80",
  "issue_name": "预付款比例过高且无担保",
  "severity": "critical",
  "priority": "P1",
  "affected_party": "buyer",
  "review_stance": "buyer",
  "legal_direction": "unfavorable_to_buyer",
  "negotiation_role": "must_fix",
  "evidence_clause_ids": ["payment_9_2"],
  "risk_reason": "甲方在乙方未交付前支付1228万元，且无预付款保函或履约保函",
  "recommendation": "降低预付款比例或增加等额担保"
}
```

重点是增加：

- `affected_party`：实际受影响方；
- `review_stance`：审查立场；
- `legal_direction`：对当前立场有利还是不利；
- `negotiation_role`：风险、优势、底线筹码、交换筹码、响应筹码。

### 3.3 法务裁决层

目标：将“检测到的事实”转成“站在当前立场下的法律和商业判断”。

示例：

```text
事实：合同未约定乙方责任上限。
买方视角：对甲方有利，应定性为底线筹码 / 响应筹码，不主动提出修改。
卖方视角：对卖方不利，应定性为高风险，建议设置合理责任上限。
```

裁决层必须比渲染层权威。LLM₂ 只能解释裁决，不能推翻裁决。

### 3.4 BN 数据与交叉校验层

目标：让 BN 成为量化辅助和冲突发现器，而不是替代法务判断。

建议为 BN 输出增加解释字段：

```json
{
  "bn_variable": "liability_cap",
  "raw_effect": "财务暴露风险下降11.4%",
  "legal_interpretation": "买方视角下，当前无责任上限是优势；该数据说明该条款价值高，应作为防守筹码而非主动修改目标",
  "report_usage": "可引用为筹码价值说明，不可写成主动新增责任上限建议"
}
```

BN 输出分三类：

1. 可直接引用：维度级概率、反事实降幅、乘数效应；
2. 需要解释：责任上限、损害赔偿暴露、管辖公平性；
3. 需要人工复核：与 Dossier 裁决冲突、与审查立场冲突、依赖大量先验概率。

### 3.5 多形态报告生成层

目标：同一 Dossier 输出不同用途的报告。

建议先实现五类输出：

1. **管理层摘要版**：1-2 页，只回答能不能签、卡在哪、必须改什么；
2. **法务审查详版**：完整逐条款分析、法律依据、修改建议；
3. **谈判作战手册**：筹码分类、对手话术、退让阶梯、底线；
4. **合同修订清单**：可直接复制进合同的替换条款；
5. **BN / 方法论附录**：数字来源、反事实、冲突与人工复核项。

### 3.6 自动评测与版本对比层

目标：让每次报告迭代可以被量化验证。

建议评分维度：

| 维度 | 分值 | 说明 |
|---|---:|---|
| 核心风险识别 | 25 | 是否抓住真正致命问题 |
| 立场正确性 | 20 | 是否正确站在买方/卖方判断 |
| 原文证据绑定 | 15 | 是否引用准确条款 |
| 修改建议质量 | 15 | 是否可直接落地 |
| 谈判策略质量 | 15 | 是否有筹码意识和退让阶梯 |
| BN 使用合理性 | 5 | 是否正确解释数据 |
| 表达与结构 | 5 | 是否清晰可读 |

## 4. 个人项目如何建立内部自动评测集

个人项目没有法务团队，也可以建立“可迭代的弱监督评测集”。核心原则是：不要一开始追求完美标准答案，而是建立可复核、可修正、可积累的 gold set。

### 4.1 从少量黄金样本开始

先选 3-5 份典型合同，不要一开始做大：

1. 买卖合同：当前这份可作为第一个 golden case；
2. 服务合同；
3. 租赁合同；
4. 技术开发或软件合同；
5. 保密协议。

每份合同只维护最关键的 10-20 个标准点。

### 4.2 标准答案不需要一开始很法律化

标准答案可以先用“事实 + 方向 + 期望行为”的形式。

以买卖合同为例：

```yaml
case_id: sales_contract_001
review_stance: buyer
must_find:
  - id: prepayment_80_no_guarantee
    expected: 识别80%预付款且无担保为致命/高风险
    evidence_keywords: ["80%", "预付款", "12280000"]
    severity_min: high
  - id: warranty_deposit_inversion
    expected: 识别质保金/保证金提交晚于预付款，构成担保结构倒挂
    evidence_keywords: ["质量保证金", "支付剩余款项前", "预付款"]
    severity_min: high
  - id: deemed_delivery_risk
    expected: 识别初验签字即视为交付导致风险转移过早
    evidence_keywords: ["初验收", "签字确认", "视为乙方已交付"]
    severity_min: high
must_not:
  - id: liability_cap_wrong_direction
    forbidden: 买方视角下，不得将乙方无责任上限直接定性为甲方风险并建议主动增加责任上限
  - id: fabricated_bn_numbers
    forbidden: 不得出现无来源概率数字
should_find_advantages:
  - 甲方住所地法院管辖
  - 未设乙方责任上限
  - 未排除间接损失
  - 发票前置付款条件
```

这类标准答案不需要法务团队才能写，因为它来自：

- 合同原文；
- 多版本报告对比；
- DeepSeek 等外部评价的交叉参考；
- 你自己反复确认过的项目规则。

### 4.3 使用“多来源共识”替代单一专家

没有专家时，可以用三类来源建立共识：

1. **合同原文事实**：金额、比例、期限、条款编号，这部分最客观；
2. **多模型交叉评价**：Claude / DeepSeek / GPT 对同一合同独立审查后，提取共同识别点；
3. **人工复核规则**：你只确认方向性问题，不需要写专业判决书。

如果三类来源都支持某个点，就可以进入 golden set。  
如果来源之间冲突，就标记为 `needs_review`，不要作为硬性评测项。

### 4.4 区分 golden case、golden pattern 与 production rule

个人项目最容易误用评测集的地方，是把“针对某份合同的标准答案”误当成“所有同类合同的业务规则”。本项目必须明确三者边界：

| 类型 | 用途 | 面向对象 | 是否用于真实审查 |
|---|---|---|---|
| `golden_cases` | 回归测试，验证系统改动后是否退步 | 固定合同样本 | 否 |
| `golden_patterns` | 抽象可泛化风险模式 | 新合同中的结构化事实 | 间接用于规则设计 |
| `production rules` / party-aware rules | 实际审查与 Dossier 裁决 | 用户上传的新合同 | 是 |

因此：

- `sales_purchase_contract_001.yaml` 中的“80%预付款”只用于测试这份买卖合同样本，不代表所有买卖合同都必须有80%预付款风险；
- 可泛化的是“高额预付款且无担保”这个 pattern；
- 真实审查时，应先抽取新合同事实，再判断是否命中 pattern 或 production rule。

正确链路：

```text
golden case 用来测试 production rules 是否稳定
golden pattern 用来沉淀可泛化判断
production rules 用来审查真实合同
```

### 4.5 自动评测先做规则评分，再做 LLM-as-judge

个人项目最稳的做法是“两层评测”，但第一层规则评分主要服务于 golden case 回归测试和 golden pattern 命中检查。

#### 第一层：规则评分

适合自动检查硬指标：

- 对固定 golden case：是否命中该合同的 `must_find`、是否触犯 `must_not`；
- 对通用 golden pattern：是否在事实条件满足时触发正确结论；
- 是否引用条款编号；
- 是否把无责任上限定性为符合当前立场的优势、风险或筹码；
- 是否出现禁止短语，例如“买方必须主动增加乙方责任上限”。

#### 第二层：LLM-as-judge

适合检查软指标：

- 分析是否深刻；
- 修改建议是否可执行；
- 谈判策略是否有交换逻辑；
- 报告是否前后矛盾；
- BN 数据解释是否克制。

但 LLM-as-judge 必须受 golden case / golden pattern 约束，不能自由制定评分标准。

### 4.6 每次人工对比都沉淀为评测资产

这次买卖合同评估已经产生了很多可沉淀资产：

- 识别 80% 预付款是硬性加分；
- 识别质保金倒挂是硬性加分；
- 将无责任上限误判为买方风险是硬性扣分；
- BN 责任上限反事实必须解释为筹码价值，不得解释为主动修改目标；
- 报告应区分风险、优势、底线筹码、交换筹码、响应筹码。

沉淀时要先判断资产类型：

- 只适用于某份合同的，写入 `golden_cases`；
- 可推广到同类交易结构的，提炼为 `golden_patterns`；
- 已足够成熟且适合真实审查的，再进入 `party_aware_rules` 或 Dossier 裁决逻辑。

## 5. 实施路线

### v2.13：立场裁决与 BN 解释稳定化

目标：先消除方向性错误。

任务：

1. 强化 party-aware 裁决规则；
2. 增加 `legal_direction` / `negotiation_role` 的一致性校验；
3. 增加 BN 解释护栏，避免将反事实误写为主动修改建议；
4. 增加渲染前自检：买方优势不得误写成买方风险。

验收：

- 买方视角下，无责任上限、未排除间接损失、甲方住所地管辖均被识别为优势/筹码；
- BN 责任上限输出不会诱导报告主动建议买方增加责任上限；
- 买卖合同 golden case 通过基础规则评测。

### v2.14：质保金倒挂与付款担保结构检测

目标：把当前买卖合同中最关键的法务洞察固化成规则。

任务：

1. 检测高额预付款；
2. 检测预付款前是否存在履约保函/预付款保函；
3. 检测质保金/保证金提交时间是否晚于预付款；
4. 生成“付款担保结构倒挂”风险项；
5. 将其纳入 Dossier 和报告模板。

验收：

- 当前买卖合同必须识别质保金倒挂；
- 报告中必须解释“先付1228万元，后收5%质保金”的结构性问题；
- 该风险不得被拆散为普通付款问题而失去优先级。

### v2.15：多形态报告输出

目标：从单一大报告升级为多用途报告包。

任务：

1. 管理层摘要版；
2. 法务详版；
3. 谈判作战版；
4. 修订条款清单；
5. BN / 方法论附录。

验收：

- 同一 Dossier 可生成至少 3 种输出；
- 管理层摘要不超过 2 页等价长度；
- 谈判作战版包含筹码矩阵、对手话术、退让阶梯、底线。

### v2.16：双层评测体系 MVP

目标：建立“golden cases 回归测试 + golden patterns 泛化规则 + production rules 对齐”的个人项目评测体系。

任务：

1. 建立 `tests/fixtures/golden_cases/`，用于固定合同样本回归测试；
2. 建立 `tests/fixtures/golden_patterns/`，用于沉淀可泛化风险模式；
3. 实现规则评分器，分别支持 case-specific 检查和 pattern 命中检查；
4. 输出评分报告，区分“固定样本退步”和“泛化模式未触发”；
5. 将成熟 pattern 对齐到 `party_aware_rules` / Dossier 裁决层；
6. 后续再接入受约束的 LLM-as-judge。

验收：

- 对当前买卖合同、销售合同 golden case 可自动给出基础回归评分；
- 能检测 `must_find` / `must_not` / `should_find_advantages`；
- 至少沉淀 3 个 golden pattern，例如高额预付款无担保、付款担保倒挂、责任上限按立场判断；
- 新合同只按命中的 pattern 检查，不套用具体 golden case 的结论。

## 6. 成功标准

短期成功标准：

- 当前买卖合同报告稳定识别 80% 预付款、质保金倒挂、初验即交付；
- 不再把买方优势误判为风险；
- BN 数字解释不再越权；
- 报告至少可拆成管理层摘要、法务详版、谈判作战版。

中期成功标准：

- 至少 3 类合同进入 golden case；
- 每次报告生成后自动评分；
- worklist / progress 能支持跨会话接续实施；
- 项目形成“生成 → 评测 → 对比 → 修正”的闭环。

长期成功标准：

- 系统不再只是报告生成器，而是可复盘、可验证、可演示的合同风控 Agent。
