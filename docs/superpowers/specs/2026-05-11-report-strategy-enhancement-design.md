# 报告策略质量增强设计方案

> 基于买卖合同 11 份报告的独立量化评分（含 DeepSeek 第三方评价对照），
> 针对新报告（contract-review-买卖合同-9.md，评分 91/100）的 5 项可改善点设计。

## 背景

v2.11 完成了 `negotiation_chip` 结构化对象端到端改造和 JSON 稳定性修复，报告-9 在此次改造后
评分达 91/100（第 2 名），与标杆报告-2（96 分）的核心差距在**谈判策略的系统性**和**报告的可读性**。

## 目标

在不改动 LLM₁ JSON schema、不新增数据模型字段、不碰 BN 推理引擎的前提下，
通过 LLM₂ prompt 增强和 dossier 模板注入，将报告策略质量从 91 分提升至 94+ 分。

## 非目标

- 不修改 `free_review_schema.py` 的数据结构
- 不新增 JSON 字段或改变 LLM₁ 输出契约
- 不重构 BN 反事实分析引擎
- 不引入新的 LLM 调用或外部依赖

---

## 设计一：跨筹码联动矩阵（B+C）

**问题**：报告-9 识别了每一枚筹码但没有告诉甲方如何用 A 换 B。
报告-2 的第五章是一张"作战地图"——量化交换比率、退让阶梯、跨筹码联动规则齐全。

**方案**：B（LLM₂ prompt 指令）+ C（dossier 模板注入）

### 改动 1：`_fmt_dossier_section()` — 模板注入

在签署建议底线之后、「合同亮点」之前，注入联动矩阵模板：

```
### 筹码联动交换矩阵（系统模板——LLM₂必须逐行填写）

以下矩阵用于第五章筹码防御分析。LLM₂必须在第五章中逐行展开此矩阵，
每行至少包含一个量化交换比率（百分比/天数/金额），禁止使用空泛话术。

| 我方筹码 | 筹码类型 | 交换对象 | 交换比率 | 退让阶梯（目标→可接受→底线） | 底线条件 |
|---------|---------|---------|---------|--------------------------|---------|
| {从 risk_items/favorable_terms 中提取筹码清单，预填前两列} | | | | | |

**填写规则：**
- "交换比率"必须量化，如"预付款每降10%，换质保金提前5日"
- "退让阶梯"必须三档：开局目标→可接受→底线
- 每个退让步骤必须附带对方需做出的对应让步
- 禁止"可考虑"、"建议律师准备"等空话
```

### 改动 2：`_build_combined_prompt()` — 第五章指令增强

将原"此章不是可选的"段落替换为：

> **第五章必须包含筹码联动交换矩阵（表格形式）。矩阵中的每一行是一个筹码对，
> 至少包含：①量化交换比率 ②三档退让阶梯 ③每步退让要求的对方对应让步。
> 禁止单向退让。**

### 文件

- `src/contract_risk_analysis/review/report_writer.py`：`_fmt_dossier_section()` + `_build_combined_prompt()`

### 验收标准

- 报告的第五章包含至少一个量化交换比率（如"预付款每降10%，换质保金尽早提交"）
- 至少一个筹码展示了三档退让阶梯
- 不存在"单向退让"（我方退让但对方不用付出任何代价）

## 设计二：反事实分析精简（A）

**问题**：BN 有 8 个反事实节点，逐一枚举导致第四章有 8 个子节，
很多效果重叠（`liability_cap` 和 `liability_cap_strength` 对财务暴露相似）。

**方案**：按效果维度分组合并

### 改动：`_fmt_dossier_section()` — BN 反事实数据段重构

当前行为：遍历所有 counterfactuals 逐一输出子节。

改为：按降幅排序，同一维度-风险组合只保留降幅最大的节点；降幅 <3% 的合并为一句话简述。
每行追加"该数字对谈判的意义"解读。

```
### BN反事实模拟数据（按改善效果排序，效果重叠项已合并）

| 改善措施 | 主要降幅维度 | 降幅 | 该数字对谈判的意义 |
|---------|------------|------|-----------------|
| 完善终止条款 | 履约交付风险 -33.7% | 法律可执行性 -21.2% | 补充几行终止条款，发货争议风险降1/3 |
| 改善解除权平衡性 | 条款失衡风险 -20.6% | — | 加对等解除权条款，合同失衡风险降1/5 |

**其他次要改善项（降幅<3%）：** 管辖公平性（-3.5%）、争议解决明确性（-3.4%）效果有限，
当前管辖约定已对甲方有利，不建议为此消耗谈判筹码。
```

**合并规则**：
1. 按 `delta_high_risk` 降序排列
2. 主维度取 `dimension_deltas` 中 `delta` 最大的维度
3. 降幅 ≥5%：独立成行 + "谈判意义"解读
4. 降幅 <3%：合并为"其他次要改善项"
5. 降幅 3%-5%：列出但简化

### 文件

- `src/contract_risk_analysis/review/report_writer.py`：`_fmt_dossier_section()` BN 反事实段

### 验收标准

- 当 BN 产出 ≥6 个反事实时，第四节的反事实项 ≤5 个独立行
- 每个独立行附带一句"对谈判的意义"

## 设计三：LLM₁ 策略思维注入（E·轻量版）

**问题**：LLM₂ 策略质量受限于 LLM₁ 的输出——如果 LLM₁ 的 `strategy` 字段只写"降到50%"，
LLM₂ 就缺少跨条款联动的素材。

**方案**：在 `_build_chip_instruction()` 末尾追加一句策略提示，
不改 JSON schema、不新增字段、不影响解析。

### 改动：`_build_chip_instruction()` 返回值末尾追加

```python
"对每个筹码，请评估其与本合同中其他条款的博弈关联——"
"是否存在用此筹码换取其他条款让步的联动空间？"
"如有，在 strategy 字段中简要提及交换方向和条件。\n"
```

### 文件

- `src/contract_risk_analysis/review/ai_review.py`：`_build_chip_instruction()`

### 验收标准

- LLM₁ 产出的 `strategy` 字段出现跨条款联动描述的频率提升
- JSON 解析成功率不下降

## 设计四：商业语言温度（C）

**问题**：报告-9 的文风偏"法律批判"，对抗性强。
报告-2 的攻击性建议外面包了一层"为了合作顺利"的糖衣，更易被对方接受。

**方案**：`_combined_system_prompt()` 中追加一条行为准则。

### 改动：`_combined_system_prompt()` 核心行为准则追加第 6 条

```python
f"6. 你的每一句修改建议，请在攻击性结论外面包一层'为了合作顺利'的包装——"
f"先承认对方的合理关切，再提出我方的修改要求。报告的最终目标是促成一份对{party_label}"
f"有利且对方愿意签署的合同，而不是把对方骂走。\n"
```

### 文件

- `src/contract_risk_analysis/review/report_writer.py`：`_combined_system_prompt()`

### 验收标准

- 报告中减少"显失公平""不合理""无法接受"等高对抗性措辞
- 增加"理解贵司的…""为了双方合作…"等桥接性措辞

## 设计五：章节一致性自检（D·prompt 自检）

**问题**：报告-7 出现了"第 5 章称为响应筹码/等鱼上钩，第 6 章却列为签署条件"的矛盾。
这本质上是 LLM 推理任务，硬规则校验会误杀合理的策略设计。

**方案**：在 `_build_combined_prompt()` 末尾增加一条自检指令，
让 LLM₂ 在生成报告后自行核对。

### 改动：`_build_combined_prompt()` 末尾，"严禁行为"之前追加

> **第五章与第六章一致性自检（必须在生成报告后逐条验证）：**
> - 第五章中归类为"底线筹码"的条款，如果在第六章签署条件中出现，
>   必须在策略中明确标注"仅在对方给出极高对价时考虑交换"，否则构成矛盾
> - 第五章中归类为"响应筹码"的条款，不得在第六章列为"签署前必须修改的条件"
> - 如果发现上述矛盾，在报告末尾「渲染器备注」中上报

### 文件

- `src/contract_risk_analysis/review/report_writer.py`：`_build_combined_prompt()`

### 验收标准

- 新报告中不再出现同一条款在第 5 章和第 6 章的战略定位矛盾

| 文件 | 改动点 | 新增行数 | 风险 |
|------|--------|---------|------|

| `report_writer.py` - `_combined_system_prompt()` | 商业语言温度 | ~3 行 | 极低 |
| `ai_review.py` - `_build_chip_instruction()` | 策略思维提示 | ~3 行 | 极低 |

不碰 schema、不碰 JSON、不碰 BN、不碰 adjudicate。
