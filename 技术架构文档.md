## 1. 系统概述

### 1.1 定位

ContractLens 是一个 **LLM + 贝叶斯网络（BN）融合** 的合同风险智能审查系统。核心差异化在于：**LLM 负责语义理解，BN 负责概率推理与反事实模拟**，两者互补而非替代。

### 1.2 与传统方案对比

| | 纯 LLM 方案 | 纯规则方案 | ContractLens |
|---|---|---|---|
| 语义理解 | 强 | 弱 | 强（LLM） |
| 可解释性 | 弱（黑箱打分） | 强 | 强（BN 推理链） |
| 反事实模拟 | 不支持 | 不支持 | 支持（"修掉 X 条款，风险降多少"） |
| 不确定性量化 | 弱（无概率模型） | 无 | 强（pgmpy 变量消元） |
| 跨维度一致性 | 弱（各维度独立打分） | 中 | 强（BN 图结构约束） |
| 幻觉风险 | 高 | 无 | 低（BN 校验 + 质量门禁） |

### 1.3 核心创新

1. **BN 作为校验器，不是打分器**：LLM₁ 自由审查 → BN 检查跨维度一致性 → LLM₂ 生成报告
2. **反事实模拟**：对每个风险项回答"如果改善这个条款，整体风险概率从 X% 降到 Y%"
3. **数据驱动的 CPT**：BN 的条件概率表由 CUAD（510 份合同）和 ContractNLI 数据集统计校准

### 3.2 v1 管线（兼容保留）

v1 管线仍可通过 `generation_mode=v1_legacy` 参数调用，流程为：LLM 约束提取 → 证据映射 → BN 推理 → LLM 润色。v1 管线中 LLM 按 BN 节点清单进行结构化提取，灵活性较低。

## 4. 贝叶斯网络引擎

### 4.1 网络结构

配置文件：`config/bayesian_network_v2.json`（~47KB，150+ 节点）

```
                         ┌──────────────────────┐
                         │ overall_contract_risk │  ← 目标节点
                         │   (3 states: H/M/L)   │
                         └──────────┬───────────┘
                                    │ Noisy-Max 聚合
            ┌───────────┬───────────┼───────────┬───────────┐
            ↓           ↓           ↓           ↓           ↓
    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 法律可   │ │ 财务暴露 │ │ 履约交付 │ │ 争议处置 │ │ 条款失衡 │
    │ 执行性   │ │   风险   │ │   风险   │ │   风险   │ │   风险   │
    └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘
         │            │            │            │            │
         └────────────┴────────────┴────────────┴────────────┘
                                    │
                            ┌───────┴───────┐
                            │ legal_semantics│  ← 法律语义层
                            │    (37 节点)   │
                            └───────┬───────┘
                                    │
                            ┌───────┴───────┐
                            │ contract_fact │  ← 合同事实层
                            │    (35 节点)   │
                            └───────────────┘
```

**三层结构**：
- **contract_fact 层**（35 节点）：合同中有没有这个条款？（present / missing / unfavorable）
- **legal_semantics 层**（37 节点）：条款的法律含义是什么？（来自 ContractNLI 标注）
- **dimension 层**（5 节点）：聚合为五大风险维度
- **overall_contract_risk**：通过 Noisy-Max 模型聚合为整体风险

### 4.2 推理引擎

使用 **pgmpy** 库（Python 贝叶斯网络框架）：

| 组件 | 实现 |
|---|---|
| 模型构建 | `pgmpy_adapter.build_model()` — 从 JSON 配置构建 `BayesianNetwork` |
| 精确推理 | `VariableElimination` — 变量消元法，适用于当前规模的离散网络 |
| 反事实模拟 | `run_sensitivity_analysis()` — 逐个节点切换状态计算 P(high) delta |
| CPT 生成 | 支持 TabularCPD 直接加载，缺失值由 `_infer_unknown_entry()` 填充（均匀先验） |
| Noisy-Max | `noisy_or.py` — overall_contract_risk 节点的聚合 CPT 生成 |

### 4.3 CPT 参数来源

```
CUAD 数据集 (510 份合同)
    │
    ├──→ cuad_preprocess.py: 统计条款出现频率、共现关系
    │       │
    │       └──→ cpt_calibrator.py: 生成 contract_fact 层 CPT
    │
ContractNLI 数据集
    │
    └──→ 标注法律语义分类（hypothesis/entailment/contradiction）
            │
            └──→ cpt_calibrator.py: 生成 legal_semantics 层 CPT

expert_estimated（专家估计）
    └──→ 无数据集覆盖的节点，标注来源，可被后续数据替换
```

每个概率数字的来源通过 `source` 字段标注，三类合法来源：
- `cuad_empirical` — CUAD 统计
- `contractnli_empirical` — ContractNLI 统计
- `expert_estimated` — 专家估计

## 5. LLM 审查层

### 5.1 模型与角色

| 调用 | 角色设定 | 任务 |
|---|---|---|
| LLM₁ | 客户代理律师（甲方/乙方） | 自由审查合同，识别风险段、缺失条款、优势条款 |
| LLM₂ | 客户代理律师 + BN 数据分析师 | 综合 BN 反事实数据生成报告，立场锚定，法律推理 |

**立场锚定**（v2.3）：LLM 角色从"中立合同设计者"改为"客户代理律师"，报告在标题显式标注审查立场（甲方视角/乙方视角）。

### 5.2 合同类型分层路由（v2.8）

配置文件：`config/contract_type_routing.yaml`

```
合同文本输入
    │
    ├──→ 标题/首段识别: 采购合同？销售合同？煤炭合同？
    │         │
    │         └──→ 匹配 contract_types.<类型>.keywords
    │
    ├──→ 正文关键词扫描: text_triggers 机制
    │         │
    │         ├── 热值/收到基 → 补充煤炭节点
    │         ├── 保密/技术资料 → 补充保密节点
    │         └── 违约金 → 补充违约金节点
    │
    └──→ 低置信度回退:
              confidence < 0.6 → expanded_then_full 模式
              多类型并列 → 取节点并集
```

**三级节点集**：
1. `universal_core`（11 个通用核心节点）：支付、交付、验收、责任上限、终止、争议等
2. `contract_types.<类型>.nodes`：销售/采购/煤炭各自专属节点包
3. `text_triggers`：正文关键词触发的补充节点

### 5.3 公司红线配置化（v2.8）

配置文件：`config/company_redlines.yaml`

```yaml
# 结构示例
通用:
  hard_rules:           # 硬红线：不可妥协
    - no_unlimited_liability:  禁止无限责任
    - exclude_indirect_loss:   间接损失必须排除
  reasoning_hints:      # 推理指引：LLM 结合行业推理
    - liability_cap_ratio:     责任上限比例按交易规模推理

销售合同:
  hard_rules:
    - retention_not_subjective_acceptance:  质保金不依赖主观验收
```

**区分原则**：硬红线是法律或商业底线，不可违反；推理指引是指导原则，LLM 需结合具体合同标的和行业惯例自行推理，不硬编码数字。

## 9. 数据溯源链

系统中任何概率数字的完整溯源路径：

```
数据集 (CUAD/ContractNLI)
    │
    ├──→ cpt_calibrator.py: 统计 CPT 参数
    │         │
    │         └──→ bayesian_network_v2.json (source 字段标注)
    │
    ├──→ pgmpy_adapter.py: 加载配置，构建 BayesianNetwork
    │         │
    │         └──→ VariableElimination: 精确推理
    │                   │
    │                   └──→ 后验概率 P(high|evidence)
    │
    ├──→ run_sensitivity_analysis(): 反事实模拟
    │         │
    │         └──→ delta_high_risk = P(high|E) - P(high|E')
    │
    └──→ LLM₂ prompt: 注入 BN 数据，生成报告
              │
              └──→ 最终报告中的百分比数字
```

**核心约束**：链上任何一环的数字修改必须标注来源。LLM₂ prompt 中有"严禁编造数字"的约束，模型参数修改也遵循同样标准。

## 10. 部署与运行

### 10.1 环境要求

- Python 3.11+
- Node.js 20+
- MySQL 8.0+（用于报告历史管理）
- DeepSeek API Key（或其他 OpenAI 兼容 API）

### 10.2 启动流程

```bash
# 后端
python -m uvicorn backend.main:app --port 9527 --reload

# 前端
cd frontend && npm install && npm run dev

# 访问
# 前端: http://localhost:5173
# API 文档: http://localhost:9527/docs
```

### 10.3 配置项

| 文件 | 说明 |
|---|---|
| `.env` | API Key、数据库连接等敏感配置 |
| `config/bayesian_network_v2.json` | BN 图结构 + CPT 参数 |
| `config/contract_type_routing.yaml` | 合同类型路由规则 |
