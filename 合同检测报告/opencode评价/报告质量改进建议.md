## 二、改进建议（按优先级排序）

### 🔴 P0："责任上限误判"——红线规则与筹码识别的系统性冲突

**问题**：V1/V2/V3都将"缺少责任上限"定性为对甲方的致命风险，而从法务实战角度看，无限责任对甲方是**谈判筹码**而非风险。根因在于`company_redlines.yaml`中：

```yaml
通用:
  hard_rules:
    - id: no_unlimited_liability
      label: 禁止无限责任
      severity: 企业红线
```

这条红线被LLM理解为"缺少责任上限=违反红线=致命风险"，但"企业红线"(内部合规)≠"对外谈判风险"。内部风控要求有责任上限，是底线条件；但从谈判角度看，对方不知道这个红线，乙方一定会主动要求加上限——这正是甲方的谈判王牌。

**改进方案**：

1. **`company_redlines.yaml`增加`redline_type`字段**：

```yaml
通用:
  hard_rules:
    - id: no_unlimited_liability
      label: 禁止无限责任
      severity: 企业红线
      redline_type: negotiation_chip  # 新增：标识此红线在谈判中是筹码而非风险
      
    - id: exclude_indirect_loss
      label: 间接损失必须排除或限制
      severity: 企业红线
      redline_type: negotiation_chip  # 同样是筹码
      
    - id: dispute_venue_buyer
      label: 争议管辖地优先约定甲方所在地法院
      severity: 企业红线
      redline_type: defensive_keep    # 防守型：必须保留，不可让渡
```

2. **`report_writer.py`的`_build_combined_prompt()`在立场规则后增加筹码/红线区分指引**（约第631行后）：

```
### 红线规则与谈判筹码的本质区分

公司红线规则中有一部分标注为 `redline_type: negotiation_chip`。
这类规则的含义是：**这条红线在内部合规层面确实是底线（不可违反），
但从对外谈判角度看，对方不知道我方有此红线——因此它是甲方的谈判筹码。**

**报告撰写原则：**
- 对于 `negotiation_chip` 类红线：
  → 不要将其定性为"致命风险"或"高风险"
  → 应将其定性为"甲方的核心谈判筹码"，明确说明对方会主动要求修改，
    我方可以此交换对方在其他条款上的让步
  → 在风险矩阵中标记为"🟡 筹码（内部红线）"而非"🔴 致命"
  → 在第五章筹码防御分析中将其列为核心筹码
  
- 对于 `defensive_keep` 类红线：
  → 定性为"有利条款"，坚决保留
  → 在谈判策略中标注为"不可让渡"

- 无论哪种类型，最终签署建议中均需满足红线要求
```

3. **`ai_review.py`的LLM₁提示词（约第472行）同步修改筹码识别指引**：

在第7条"筹码识别"后追加：

```
7.5 **红线规则的双重角色识别**：如果合同缺少某项条款恰好对应公司红线规则
（如责任上限缺失对应"禁止无限责任"红线），请特别注意：
该缺失从内部合规角度是红线（必须解决），但从对外谈判角度是甲方的筹码
（乙方不知道我方红线，会主动要求加上限）。
在negotiation_chip字段中标注此双重角色，而不要将其简单定性为"致命风险"。
```

### 🔴 P0：筹码识别模块缺乏" adversarial thinking"

**问题**：V3虽然列举了甲方有利条款（4项✅），但未像V0那样将其转化为动态交换筹码。V0识别了4项筹码并给出了对方攻击话术和三层回应，V3只列了有利条款。

**根因**：`report_writer.py`第657-697行的筹码防御分析指令结构是好的，但**LLM₁的`negotiation_chip`字段质量直接决定了LLM₂的筹码章质量**。当前LLM₁提示词（第472-474行）对筹码的定义过于简单：

```
"筹码是指对方有强烈动机去修改、但我方应当坚守的条款"
```

这个定义缺乏"adversarial thinking"——缺少从对方视角反向思考的引导。

**改进方案**：

1. **增强LLM₁的筹码识别提示词**（`ai_review.py`第472-474行）：

替换为：

```
7. **筹码识别（adversarial thinking）**：
   从{party_label}的视角，识别合同中的核心谈判筹码。筹码是指同时满足以下
   三个条件的条款：
   
   a) **{party_label}拥有或对{party_label}有利**：该条款赋予了{party_label}  
      某种权利或优势（如无责任上限对我方索赔有利、管辖在我方所在地对我方维权
      有利）
   
   b) **对方有强烈动机去修改**：站在对方律师的角度，他们一定会要求修改此条款
      （如对方会认为无责任上限是"不可接受的风险"而主动要求加上限）
   
   c) **我方可以用此交换对方在其他核心条款上的让步**：此条款在谈判中具有
      实际交换价值（如我方可以"同意加上责任上限"来交换"降低预付款比例"）
   
   对每个筹码项，fill negotiation_chip字段，包含：
   - 筹码条款位置（第几条）
   - 为什么它是筹码（三方分析：对谁有利、对方为何想改、我方如何利用）
   - 对方最可能从什么角度攻击（法律角度/商业角度/策略角度，给出具体攻击话术）
   - 守住该筹码的谈判策略（防守话术+交换方案+底线）
```

2. **在`report_writer.py`第664行"筹码识别"小节增加"筹码分类矩阵"**：

```
**5.1 筹码识别**

从合同的具体条款出发，识别{party_label}的1-2个**核心谈判筹码**。
对每个筹码，必须完成以下分析矩阵：

| 筹码维度 | 分析内容 |
|---------|---------|
| 条款位置 | 合同第几条 |
| 表面属性 | 该条款对谁有利（通常是表面看对对方不利的条款） |
| 为什么是筹码 | 对方有强烈动机修改它（如：无限责任→对方律师一定要求加上限） |
| 对方攻击方向 | 对方律师会用什么法律/商业理由要求修改（给出具体攻击话术） |
| 交换价值 | 我方可以用此筹码交换对方在哪些核心条款上的让步 |
| 风险提示 | 如果我方主动放弃此筹码而不要求对价，会有什么后果 |

特别注意：公司红线规则中的 negotiation_chip 类型（如责任上限缺失、间接损失未排除），
虽然在内部合规层面是底线，但在对外谈判中恰恰是甲方的核心筹码——对方不知道我方的红线，
会主动要求修改，我方应利用这一点交换其他让步。
```

### 🟠 P1："有利条款保留"策略不够果断

**问题**：V3的BN交叉校验中正确拒绝了"改善解除权平衡性"和"改善管辖安排公平性"两项建议（因为会削弱甲方优势），但V1和V2没有做到这一点。这说明LLM₂在处理"有利条款"时缺乏系统性指引。

**改进方案**：

在`report_writer.py`的`_build_combined_prompt()`第631行"立场规则"后增加：

```
### 有利条款保护规则（强制执行）

1. 如果你识别到某条款对{party_label}有利（如对方违约责任无上限、我方拥有单方解除权、
   争议管辖在我方所在地等），你必须：
   a) 在风险总览的表格中将其标记为"✅ 有利条款"
   b) 在签署建议中明确声明"此条款必须保留，不可让渡"
   c) 在谈判策略中将其列为核心防御项

2. **禁止主动让渡有利条款**：BN反事实模拟可能建议"改善XX公平性/平衡性"，
   如果该"改善"意味着削弱{party_label}的既有优势（如增加对方的解除权、改变
   对我方有利的管辖安排），你必须在交叉校验中明确标注"不采纳"，
   并说明原因：此条款是{party_label}的核心优势，改善对方的地位不在我方利益之内。

3. **有利条款的交易价值**：有利条款在谈判中的价值不仅仅是"保留"——如果对方要求
   修改，我方可以以此作为交换条件，获取对方在其他条款上的让步。
   在筹码防御章节中必须说明每个有利条款的交易价值。
```

### 🟠 P1：LLM₁的"付款与担保时间逻辑"审查要点需要增强

**问题**：AI网页报告独步全场的发现是"付款与质保金递交节点倒挂"——先付全款后收支票/保函。而V1/V2/V3都未深入分析这一结构性缺陷。

**根因**：`company_redlines.yaml`中有正确的`reasoning_hints`：

```yaml
- id: payment_guarantee_timing
  label: 质保金/履约保证金应在付款前提交
  description: "审查付款与担保的时间顺序..."
```

但这条是`reasoning_hint`而非`hard_rule`，LLM可能未充分重视。其次，`ai_review.py`第488-489行的审查要点第1条虽然提到了"付款与担保的时间顺序"，但描述偏简略。

**改进方案**：

1. **将`payment_guarantee_timing`从reasoning_hints升级为hard_rule**（至少对采购合同）：

```yaml
采购合同:
  hard_rules:
    # ... existing rules ...
    - id: payment_before_guarantee
      label: 担保必须先于付款提交
      description: "如果合同要求甲方在乙方提交履约保证金/质保金之前支付大额款项（尤其是超过50%的预付款），
        则担保对甲方形同虚设。必须在审查中标注此结构性倒挂，并计算甲方的无担保资金敞口。"
      severity: 企业红线
```

2. **增强`ai_review.py`第488-490行的审查要点**：

```
1. **付款与担保的时间逻辑**：审查付款节点与担保措施的时间顺序——
   是否存在[先付全款再收担保]的结构性倒挂？
   具体检查：
   a) 预付款比例是否超过30%（超过则标记为高风险）
   b) 质保金/履约保证金是否在付款之后才提交（如是，标记为结构性倒挂）
   c) 计算甲方在乙方未提供任何担保情况下的资金敞口（如80%预付款=1228万元，质保金仅5%=76.75万元，
      倒挂比例=76.75/1228≈6.3%）
   d) 在risk_description中明确指出此倒挂的商业后果：担保形同虚设
```

### 🟡 P2：BN反事实数据的"反建议"机制缺失

**问题**：当前BN反事实模拟只输出"改善XX→风险降低Y%"，但没有"恶化XX→风险升高Y%"的反向模拟，也没有"如果我方让渡某有利条款→风险如何变化"的情景分析。

在V3中，LLM₂通过人工判断拒绝了"改善解除权平衡性"和"改善管辖安排公平性"，但这个判断依赖LLM的法务推理能力，缺乏BN量化数据的支撑。

**改进方案**：

1. **在`bn_validator.py`的`run_counterfactual_analysis()`中增加反向模拟**：

```python
def run_counterfactual_analysis(
    self, node_states: dict[str, str], top_n: int = 8
) -> tuple[list[CounterfactualResult], list[CounterfactualResult]]:
    """Run counterfactual simulations and return both improvement and degradation results.
    
    Returns:
        (improvements, degradations): 
        improvements = what-if we improve each risk factor
        degradations = what-if the counterparty's proposed changes take effect
    """
    # ... existing improvement logic ...
    
    # NEW: For key buyer-friendly nodes, simulate what happens if we give them up
    buyer_friendly_nodes = {
        "unlimited_liability": "absent",  # if we add liability cap per counterparty request
        "buyer_termination_right": "favorable",  # if we add seller termination right
        "dispute_venue_buyer": "absent",  # if we change venue to neutral
    }
    # Run sensitivity for each buyer-friendly node with WORSENED state
    # Return as separate list "degradations"
```

2. **在`report_writer.py`的反事实数据格式化中增加反向模拟展示**：

```
### 交叉校验中的"反建议"标注

当BN模拟显示"改善XX公平性"能降低风险时，你必须同时回答：
- 改善这个维度的"公平性"是否意味着削弱{party_label}的既有优势？
- 如果是，在交叉校验判断中明确标注"不采纳"，并给出理由：
  "此改善虽能降低整体风险，但代价是削弱甲方的核心优势（如解除权/管辖权），
  从{party_label}利益最大化角度，不应采纳。"
```

### 🟡 P2：LLM₂报告第四章"反事实分析排版"导致阅读疲劳

**问题**：V3报告第四章有8-9个反事实项，每个都是"维度级主要数据+整体级辅助参考+交叉校验判断"三段式，产生了大量重复模板化文本，读者难以抓住重点。

**改进方案**：

在`report_writer.py`第576-598行的反事实数据使用规则中增加：

```
### 反事实分析的结构优化

1. **前3个反事实项**（优先级最高的）：使用完整的三段式结构（BN数据+整体参考+交叉校验判断）
2. **其余反事实项**：使用精简格式，一行展示核心数据和判断结论
3. **所有反事实项分析完毕后**，必须提供一张整合的"优先级排序表"，
   包含以下列：优先级 | 改善措施 | BN维度级降幅 | 法务判断 | 最终建议（采纳/不采纳/合并）

示例精简格式：
### N. 改善XX条款
- BN：维度级降幅X%（法律可执行性↓Y%），整体降幅Z% | 交叉校验：一致，建议采纳
```

### 🟡 P2：五维评分卡缺失

**问题**：AI网页报告和V2都有清晰的五维风险评分（如"财务暴露85%/高"），但V0、V1、V3都依赖文字描述或简单的🔴/🟠分级，缺少量化的维度级概率。

当前BN已经计算了五个维度的后验概率（`legal_enforceability_risk`, `financial_exposure_risk`, `performance_delivery_risk`, `dispute_resolution_risk`, `clause_balance_risk`），但LLM₂没有在报告的"风险总览"章节中系统展示。

**改进方案**：

在`report_writer.py`的输出格式要求（第612-614行）的"风险总览"章节中增加：

```
## 二、风险总览

### 2.1 五维风险评分卡

请首先输出以下五维风险评分卡（数据来自BN后验概率）：

| 风险维度 | P(high) | 风险等级 | 简要说明 |
|---------|---------|---------|---------|
| 财务暴露风险 | XX% | 高/中/低 | 一句话说明主要驱动因素 |
| 履约交付风险 | XX% | 高/中/低 | ... |
| 法律可执行性风险 | XX% | 高/中/低 | ... |
| 争议处置风险 | XX% | 高/中/低 | ... |
| 条款失衡风险 | XX% | 高/中/低 | ... |

风险等级划分标准：P(high)≥40%=高，20%-40%=中，<20%=低
```

同时在`_fmt_bn_validation()`函数中，确保BN posteriors的概率数据被传入LLM₂的prompt。

### 🟢 P3：报告之间的"进化追踪"机制

**问题**：V1→V2→V3的改进方向是试错性的，缺乏系统化的进化追踪。建议在报告中增加"版本变更日志"章节。

**改进方案**：

在`FreeReviewOutput`数据模型中增加版本追踪字段：

```python
@dataclass(frozen=True)
class FreeReviewOutput:
    # ... existing fields ...
    previous_version_issues: list[str] | None = None  # 上一版本识别的问题
    improvements_over_previous: list[str] | None = None  # 相比上一版本的改进
```

在LLM₁ prompt中增加：

```
如果你收到上一版的审查结果（previous_version_issues），请特别注意：
1. 上一版本是否存在将"有利缺失"误判为"风险"的情况（如将"无责任上限"判为致命风险）
2. 上一版本的筹码识别是否充分
3. 上一版本的BN数据解读是否出现了与法务判断矛盾的建议
```

## 三、改进优先级总结

| 优先级 | 改进项 | 影响范围 | 预期效果 |
|-------|--------|---------|---------|
| 🔴 P0 | 红线规则与筹码区分 (`company_redlines.yaml` + prompt) | 所有报告 | 消除"责任上限=致命风险"的系统性误判 |
| 🔴 P0 | 筹码识别adversarial thinking增强 (`ai_review.py` + `report_writer.py`) | 所有报告 | 筹码质量从V3级别提升至V0级别 |
