# BN-Contract-Risk-Analysis 分阶段待办清单

## 1. 使用说明

本清单面向推荐方向：

> 将项目演进为“高可解释、可评估、真正概率化”的合同风险推理系统。

清单分为四个阶段：

- 阶段 0：战略定标
- 阶段 1：推理核心升级
- 阶段 2：评估与解释闭环
- 阶段 3：产品化增强

建议按顺序推进，不要过早进入前端包装和产品功能堆叠。

---

## 2. 阶段 0：战略定标

### 目标

统一项目目标，避免后续开发发散。

### 待办

- [ ] 明确项目主线定位为“研究型解释引擎”，而非纯展示型 demo
- [ ] 明确当前版本的研究问题：为什么 LLM+BN 比纯 LLM 更适合合同风险评估
- [ ] 明确近期关注的合同类型范围（建议先聚焦 NDA 与服务类合同）
- [ ] 明确核心风险维度是否固定为 5 大类，还是后续按合同类型扩展
- [ ] 明确阶段性成功标准：
  - [ ] 可复现实验
  - [ ] 节点级解释

### 阶段产出

- 一份统一目标说明
- 一份核心研究问题定义
- 一份版本 1 范围边界说明

## 3. 阶段 1：推理核心升级

### 目标

将当前“规则加权风险层”升级为“真实概率推理层”。

### 1. 节点与状态重构

- [ ] 盘点当前所有 node，按层级重组：
  - [ ] 合同事实层
  - [ ] 法律语义层
  - [ ] 风险维度层
  - [ ] 决策层
- [ ] 为每个 node 写清定义、含义、取值空间、示例
- [ ] 重构状态语义，至少区分：
  - [ ] `present`
  - [ ] `missing`
  - [ ] `unfavorable`
  - [ ] `acceptable`
  - [ ] `ambiguous`
  - [ ] `unknown`

### 2. 证据层升级

- [ ] 将当前 `build_evidence.py` 从简单映射升级为独立 evidence 模块
- [ ] 为 evidence item 增加：
  - [ ] 来源 chunk / paragraph
  - [ ] 原文 span
  - [ ] confidence
  - [ ] ambiguity flag
  - [ ] supporting facts
- [ ] 增加 evidence conflict detection
- [ ] 增加 unknown 与 missing 的区分规则

### 3. BN 推理升级

- [ ] 评估并选择推理库（如 `pgmpy`）
- [ ] 将 JSON 配置重构为真正支持 CPT 与图结构的格式
- [ ] 为节点关系建立显式边
- [ ] 支持 evidence 注入与 posterior 更新
- [ ] 输出节点后验概率而不是仅输出打分
- [ ] 增加敏感性分析能力

### 4. 参数治理

- [ ] 明确每个 CPT/weight 的来源：
  - [ ] 专家先验
  - [ ] 数据统计
  - [ ] 混合校准
- [ ] 建立 config versioning 机制
- [ ] 增加配置校验工具

### 阶段产出

- 一版真正的 BN 推理模块
- 一版节点/状态定义文档
- 一版 evidence 规范
- 一版参数来源说明

## 4. 阶段 2：评估与解释闭环

### 目标

让系统从“能输出结果”升级为“能验证结果、解释结果、持续优化结果”。

### 1. 上游抽取评估

- [ ] 接入 `ContractNLI`、`CUAD` 或自建子集
- [ ] 建立 extraction benchmark
- [ ] 评估 clause_type 识别准确率
- [ ] 评估 status 判断准确率
- [ ] 评估 finding_key 触发准确率
- [ ] 做错误分布分析

### 2. 中游映射评估

- [ ] 建立 finding → node 的 gold mapping 数据集
- [ ] 评估 evidence mapping 准确率
- [ ] 统计误映射、高风险误触发、漏触发情况

### 3. 风险推理评估

- [ ] 建立合同级风险标注样本
- [ ] 定义评估指标：
  - [ ] overall risk 一致率
  - [ ] dimension risk 一致率
  - [ ] top risks 一致率
  - [ ] manual review 召回率
- [ ] 对比 baseline：
  - [ ] pure LLM
  - [ ] LLM + rules
  - [ ] LLM + BN
  - [ ] LLM + BN + explanation LLM

### 4. 解释层升级

- [ ] 将当前 `report_writer.py` 从润色器升级为结构化解释器
- [ ] 输出基于节点的因果说明
- [ ] 输出“关键风险驱动因素”
- [ ] 输出“修订后预期风险下降点”
- [ ] 支持法务版/业务版/管理层版报告模板

### 5. 实验与反馈记录

- [ ] 记录 prompt 版本
- [ ] 记录 evidence mapping 版本
- [ ] 记录 BN config 版本
- [ ] 记录 benchmark 输出结果
- [ ] 设计人工反馈录入入口

### 阶段产出

- 一套 benchmark 体系
- 一套 baseline 对比结果
- 一版解释增强报告
- 一套版本化实验记录机制

## 8. 最终提醒

项目的真正价值不在于页面做得多复杂，也不在于模型名字换得多快，而在于：

> 你能否构建出一套“比纯 LLM 更可信、更可解释、更可评估”的合同风险推理系统。

因此，开发顺序一定要围绕“推理核心”和“评估闭环”展开。
