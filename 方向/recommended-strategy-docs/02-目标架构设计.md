# BN-Contract-Risk-Analysis 目标架构设计

## 1. 架构目标

目标架构不应仅仅是“LLM 读合同，再输出结果”，而应演进为：

> LLM 提取事实 → Evidence Layer 形成标准化证据 → True BN 进行概率推理 → Explanation LLM 生成结构化解释与建议

也就是：

> **Contract Text -> Extraction LLM -> Evidence Layer -> True Bayesian Network -> Explanation LLM -> Multi-role Output**

该架构的核心思想是：

- 上游 LLM 负责识别和抽取
- 中间 Evidence Layer 负责规范化和对齐语义
- 真正的 BN 负责风险传播、聚合与后验更新
- 下游 LLM 负责解释、沟通和建议生成

---

## 2. 当前架构与目标架构的差异

### 当前架构

当前项目实际流程为：

1. `ai_review.py` 从合同文本抽取 findings
2. `build_evidence.py` 将 findings 映射为 node states
3. `bn/inference.py` 基于规则和权重进行风险打分
4. `report_writer.py` 对结果进行文本润色

这套架构的问题不在于“不能用”，而在于：

- 中间推理层偏规则打分，不是真正概率图推理
- 证据层语义不够完整
- 下游 LLM 的解释能力尚未充分发挥

### 目标架构

目标架构需要让每一层职责更清晰：

1. Extraction Layer：抽取合同事实
2. Evidence Layer：标准化、去歧义、携带不确定性
3. Bayesian Reasoning Layer：显式概率推理
4. Explanation Layer：面向不同角色生成解释与建议
5. Evaluation & Feedback Layer：支撑持续优化

---

## 3. 分层设计

### 3.1 Contract Ingestion Layer

职责：

- 接收原始合同文本、文件、段落信息
- 做基础预处理
- 提取 metadata（合同编号、合同类型、来源、语言等）

建议能力：

- 文本切段与 chunking
- 页码/段落号保留
- 文本标准化
- 文档来源记录

输出：

- `ContractDocument`
- `ContractChunk[]`

### 3.2 Extraction LLM Layer

职责：

- 将合同文本转化为结构化合同事实与初始风险信号

建议抽取内容不仅包括当前的 `finding`，还包括：

- clause_type
- clause_span
- source_chunk_id
- obligation_party
- benefit_party
- monetary_limit
- time_limit
- trigger_condition
- extracted_fact_type
- extraction_confidence
- ambiguity_flag

输出：

- `ExtractedClauseFact[]`
- `ExtractionIssue[]`

### 3.3 Evidence Layer

职责：

- 将抽取事实映射为统一证据对象
- 区分“未识别”“确实缺失”“表达模糊”“存在但不利”等不同语义
- 为 BN 提供标准化输入

建议 evidence 的状态体系至少支持：

- `present`
- `missing`
- `unfavorable`
- `acceptable`
- `ambiguous`
- `unknown`

建议输出：

- `RiskEvidenceItem[]`
- `NodeObservation[]`
- `ObservationConfidence`

注意：

Evidence Layer 是未来系统的关键层。它是从“LLM 输出”过渡到“推理输入”的桥梁，必须比当前简单映射更强。

### 3.4 True Bayesian Reasoning Layer

职责：

- 根据节点、边、先验、条件概率进行真实推理
- 根据 observed evidence 更新后验概率
- 产出 node posterior、dimension posterior、overall posterior

建议采用：

- `pgmpy` 或同类库
- 明确 node graph
- 明确 CPT 来源与版本

建议节点分层：

#### 合同事实层
- 是否存在终止条款
- 是否存在责任上限
- 是否存在争议解决条款

#### 法律语义层
- 条款完整性
- 权利义务平衡性
- 管辖公平性
- 解释确定性

#### 风险维度层
- 法律可执行性风险
- 财务暴露风险
- 履约交付风险
- 争议处置风险
- 条款失衡风险

#### 决策层
- overall_contract_risk
- signing_recommendation
- requires_manual_review

输出：

- `PosteriorNodeState[]`
- `DimensionRiskDistribution[]`
- `OverallRiskDistribution`
- `RiskContributionTrace`

### 3.5 Explanation LLM Layer

职责：

- 将 BN 的结构化推理结果转换为人类可读解释
- 输出针对不同角色的风险说明与建议

建议模式：

#### 面向法务
- 条款级风险因果链
- 建议补齐内容
- 建议修订方向

#### 面向业务
- 风险影响
- 是否影响签署
- 优先谈判项

#### 面向管理层
- 总体风险摘要
- 是否建议签署
- 最主要的 3 个风险驱动因素

输出：

- `LegalReport`
- `BusinessReport`
- `ExecutiveSummary`
- `NegotiationSuggestions`

### 3.6 Evaluation & Feedback Layer

职责：

- 记录抽取、映射、推理、解释全过程指标
- 支持人工反馈反哺
- 支持 prompt、mapping、BN config 版本评估

建议能力：

- benchmark runner
- error analysis
- gold dataset comparison
- reviewer feedback ingestion

---

## 4. 目标数据流

目标数据流建议为：

1. 输入合同文本
2. 切分文本并保留段落位置
3. 上游 LLM 抽取 clause facts
4. Evidence Layer 将 facts 规范化为节点观察值
5. BN 根据 observation 更新后验概率
6. 系统计算关键风险驱动因子与人工复核建议
7. 下游 LLM 生成角色化风险解释与建议
8. 评估层记录本次运行的模型版本、配置版本、结果和反馈

```text

  extraction/
  evidence/
  bn/
  explanation/
  evaluation/

  web/
```

其中：

- `extraction/` 专注上游 LLM 抽取
- `evidence/` 专注证据标准化与对齐
- `bn/` 专注概率推理
- `explanation/` 专注下游解释生成
- `evaluation/` 专注实验与指标

## 6. 关键设计原则

### 原则一：LLM 不直接决定最终总体风险

最终总体风险应由显式推理层决定，而不是由 LLM 自由生成。

### 原则二：Evidence Layer 必须独立

不要把 LLM 输出直接塞进 BN。中间必须有规范化语义层。

### 原则三：风险结论必须可回溯

每个总体风险结论，都应该能够追溯到：

- 哪些 clause facts
- 哪些 node observations
- 哪些 risk dimensions

### 原则四：不确定性必须保留

不要把所有不明确情况都粗暴视为缺失或高风险。

### 原则五：解释层不能覆盖推理层

Explanation LLM 负责说明，不负责篡改概率结果。

---

## 7. 最终目标形态

理想状态下，系统应能回答这样的问题：

- 为什么总体风险是中风险而不是低风险？
- 是哪些证据导致财务暴露风险升高？
- 哪个节点对结果影响最大？
- 如果补充终止条款，总体风险会下降多少？
- 哪些问题必须升级人工法务复核？

当系统具备这些能力时，它才真正从“合同审查 Demo”进化为“合同风险推理系统”。
