## 2. 使用范围

本版本主要服务于阶段一的第一批重构工作，因此遵循以下约束：

1. 优先支持当前项目中最核心的合同风险问题
2. 优先兼容现有 NDA / 通用合同审查路径
3. 优先支持中间层重构，不追求一次覆盖所有合同类型
4. 第一版尽量少而稳，不追求节点数量过多

因此，本文档中的节点分为：

- `P0`：第一批必须落地的核心节点
- `P1`：阶段一建议落地的扩展节点
- `P2`：后续阶段再扩展的节点

## 3. 状态字典（State Dictionary）

### 3.1 通用状态全集

建议第一版采用以下 6 个基础状态语义：

- `present`
- `missing`
- `acceptable`
- `unfavorable`
- `ambiguous`
- `unknown`

不是所有 node 都需要全部 6 个状态，但所有 node 的状态都应尽量从这套基础语义中派生。

---

### 3.2 状态定义

#### `present`

中文含义：存在

定义：

- 表示相关条款、事实、义务、安排在文本中被明确识别到

适用层：

- 合同事实层

示例：

- 文本明确写明“任一方可提前 30 日书面通知终止合同”

---

#### `missing`

中文含义：缺失

定义：

- 在当前文本覆盖和审查上下文下，可以较有把握地认为该条款或事实不存在

适用层：

- 合同事实层

示例：

- 合同全文中未发现争议解决条款

注意：

- `missing` 不等于 `unknown`

---

#### `acceptable`

中文含义：可接受

定义：

- 表示该法律语义项或商业安排对我方风险总体可控，没有明显失衡或重大不利

适用层：

- 法律语义层

示例：

- 责任上限为合同总价，且排除了间接损失

---

#### `unfavorable`

中文含义：不利

定义：

- 表示该项安排明显偏向对方，或者显著放大我方财务、法律、履约风险

适用层：

- 法律语义层

示例：

- 无限责任
- 单方解除权只赋予对方

---

#### `ambiguous`

中文含义：模糊

定义：

- 文本中存在相关内容，但表达不稳定、边界不清、解释空间过大，难以稳定判断

适用层：

- 合同事实层
- 法律语义层

示例：

- 写了“合理期限内通知”，但没有说明具体天数

---

#### `unknown`

中文含义：未知

定义：

- 当前信息不足，无法做出有效判断；不能据此直接推定缺失或高风险

适用层：

- 合同事实层
- 法律语义层

示例：

- 文本不完整
- 合同附件未上传
- LLM 对该问题未返回稳定结果

---

### 3.3 状态判定顺序

建议状态判定顺序如下：

1. 先判断有没有足够信息
   - 信息不足 → `unknown`
2. 再判断是否存在相关条款/事实
   - 明确不存在 → `missing`
   - 明确存在 → `present`
   - 疑似存在但不稳定 → `ambiguous`
3. 对法律语义层再判断质量与倾向性
   - 合理可控 → `acceptable`
   - 明显不利 → `unfavorable`
   - 条款模糊 → `ambiguous`
   - 无法判断 → `unknown`

---

### 3.4 状态冲突处理原则

如果同一 node 被多条 evidence 命中且状态冲突，建议采用：

1. 高置信度优先于低置信度
2. 如冲突无法消解，落为 `ambiguous`
3. 保留 supporting evidence，不直接抹掉冲突

这意味着未来不要再用“单一覆盖式替换”处理所有冲突。

## 4. 第一版节点总览

第一版节点共分四层：

1. 合同事实层
2. 法律语义层
3. 风险维度层

## 5. 第一层：合同事实层节点

这一层只负责表达“是否存在某个条款/事实”。

### 5.1 P0 节点

#### 1. `termination_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在终止条款
- 典型证据：终止条件、终止通知、提前终止约定
- 优先级：P0

#### 2. `liability_cap_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在责任上限条款
- 典型证据：赔偿责任上限、责任限制、免责边界
- 优先级：P0

#### 3. `governing_law_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在适用法律条款
- 典型证据：适用中华人民共和国法律、适用加州法律等
- 优先级：P0

#### 4. `dispute_resolution_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在争议解决条款
- 典型证据：仲裁、诉讼、管辖法院、仲裁委员会
- 优先级：P0

#### 5. `confidentiality_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在保密义务条款
- 典型证据：保密信息定义、保密义务、例外情形
- 优先级：P0

### 5.2 P1 节点

#### 6. `acceptance_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在验收条款
- 优先级：P1

#### 7. `payment_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在付款条款
- 优先级：P1

#### 8. `delivery_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在交付条款
- 优先级：P1

#### 9. `ip_ownership_clause_existence`

- 层级：合同事实层
- 状态空间：`present` / `missing` / `ambiguous` / `unknown`
- 含义：是否存在知识产权归属条款
- 优先级：P1

## 6. 第二层：法律语义层节点

这一层表达“条款质量、倾向性、可执行性”。

### 6.1 P0 节点

#### 1. `termination_clause_completeness`

- 层级：法律语义层
- 父节点：`termination_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：终止条款是否完整，是否覆盖终止条件、通知期、终止后责任
- 优先级：P0

#### 2. `termination_right_balance`

- 层级：法律语义层
- 父节点：`termination_clause_completeness`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：终止权是否平衡，是否仅对一方有利
- 优先级：P0

#### 3. `liability_cap_strength`

- 层级：法律语义层
- 父节点：`liability_cap_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：责任上限安排是否合理
- 优先级：P0

#### 4. `governing_law_clarity`

- 层级：法律语义层
- 父节点：`governing_law_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：适用法律是否明确、稳定
- 优先级：P0

#### 5. `dispute_resolution_clarity`

- 层级：法律语义层
- 父节点：`dispute_resolution_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：争议解决方式是否明确可执行
- 优先级：P0

#### 6. `jurisdiction_fairness`

- 层级：法律语义层
- 父节点：`dispute_resolution_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：争议解决地点或管辖安排是否公平
- 优先级：P0

#### 7. `confidentiality_scope_reasonableness`

- 层级：法律语义层
- 父节点：`confidentiality_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：保密范围是否合理，是否过宽或过窄
- 优先级：P0

### 6.2 P1 节点

#### 8. `damages_exposure`

- 层级：法律语义层
- 父节点：`liability_cap_strength`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：损害赔偿暴露程度是否过高
- 优先级：P1

#### 9. `acceptance_process_clarity`

- 层级：法律语义层
- 父节点：`acceptance_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：验收流程是否明确
- 优先级：P1

#### 10. `payment_obligation_balance`

- 层级：法律语义层
- 父节点：`payment_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：付款安排是否失衡
- 优先级：P1

#### 11. `delivery_obligation_clarity`

- 层级：法律语义层
- 父节点：`delivery_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：交付要求、标准、时限是否明确
- 优先级：P1

#### 12. `ip_ownership_clarity`

- 层级：法律语义层
- 父节点：`ip_ownership_clause_existence`
- 状态空间：`acceptable` / `unfavorable` / `ambiguous` / `unknown`
- 含义：知识产权归属是否明确且对我方可接受
- 优先级：P1

## 10. 节点配置结构建议

建议未来以独立配置文件持有 node schema。每个节点至少包含：

- `node_name`
- `node_layer`
- `description`
- `states`
- `parents`
- `priority`

建议示例：

```json
{
  "node_name": "termination_clause_completeness",
  "node_layer": "legal_semantics",
  "description": "终止条款是否覆盖终止条件、通知期和终止后义务",
  "states": ["acceptable", "unfavorable", "ambiguous", "unknown"],
  "parents": ["termination_clause_existence"],
  "priority": "P0"
}
```

## 11. 与第一批实施的直接关系

本文档落地后，第一批代码修改应直接围绕下列工作展开：

1. 将 P0 node 写入配置
2. 将 finding 先映射为 `EvidenceItem`
3. 再将 `EvidenceItem` 聚合为 `NodeObservation`
4. 让推理层读取 `NodeObservation`
5. 在保留旧逻辑兼容的前提下，先打通 P0 路径

换句话说：

> 本文档不是纯说明文，它将直接决定第一批中间层重构的边界和数据结构。

如果没有节点清单与状态字典，后续重构会不断摇摆；如果这份文档定义稳定，阶段一的实现就会明显顺畅。

一句话总结：

> 第一版 node 不求多，但必须语义清晰、层级明确、能直接支撑 evidence 与 observation 的落地。
