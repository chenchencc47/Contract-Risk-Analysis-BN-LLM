# BN-Contract-Risk-Analysis 阶段一实施方案

## 1. 文档目标

本实施方案对应路线图中的第一阶段：

> 从 Demo 升级为可信研究原型

对应待办清单中的重点方向：

- 节点与状态重构
- 证据层升级
- BN 推理升级
- 参数与配置治理

本方案的目的，是把这些方向转成可以实际推进的实施步骤，而不是停留在抽象建议层面。

---

## 2. 阶段一目标定义

阶段一不追求把整个系统一次性产品化，而是聚焦一个明确目标：

> 让项目从“规则驱动的风险 demo”升级为“具备真实推理结构、规范中间语义、可做基线对比的研究原型”。

### 本阶段完成后，系统应达到以下状态

1. 拥有清晰的 node 分层与 state 语义定义
2. 拥有独立的 Evidence Layer
3. 拥有从 finding 到 observation 的标准映射链
4. 拥有可替换为真实 BN 推理引擎的中间结构
5. 拥有最小可用 benchmark 与对比框架

---

## 3. 阶段一边界

### 本阶段重点做什么

- 做实中间层
- 稳住推理输入结构
- 为真实 BN 推理打基础
- 建立最小评估能力

### 本阶段暂时不重点做什么

- 不优先做复杂前端升级
- 不优先做多合同类型全面扩展
- 不优先做完整产品权限系统
- 不优先做大规模部署系统

这样做是为了确保项目资源集中在真正的壁垒上。

---

## 4. 实施顺序总览

建议将阶段一拆成五个子阶段推进。

### 子阶段 1：定义冻结

目标：冻结 node、state、evidence 的基础设计。

### 子阶段 2：证据层重构

目标：把当前 `build_evidence.py` 拆成可维护的 evidence 模块。

### 子阶段 3：推理层替换准备

目标：让现有系统支持新的 observation 结构，并为真实 BN 推理留接口。

### 子阶段 4：评估框架建立

目标：建立最小 benchmark 与基线对比。

### 子阶段 5：端到端跑通

目标：让新中间层在当前 demo/API/report 流程下可跑通。

---

## 5. 子阶段 1：定义冻结

### 目标

在真正改代码前，先冻结中间层语义，不然重构会反复摇摆。

### 任务清单

- [ ] 建立 node 分层清单
- [ ] 为每个 node 写说明、父节点、状态空间
- [ ] 统一六类状态语义定义
- [ ] 明确不同合同类型下哪些 node 是公共节点，哪些是特定节点
- [ ] 定义 `EvidenceItem` 字段
- [ ] 定义 `NodeObservation` 字段
- [ ] 定义 finding → evidence → observation 的标准流程

### 建议产出物

- `node_schema_v1`
- `state_semantics_v1`
- `evidence_schema_v1`
- `observation_schema_v1`

### 完成标准

- 所有核心 node 有书面定义
- 所有状态有明确判定规则
- 不再依赖口头理解推进后续开发

---

## 6. 子阶段 2：证据层重构

### 目标

让当前 `findings -> node_states` 的直接映射，升级为：

> `findings -> EvidenceItem[] -> NodeObservation[]`

### 任务清单

- [ ] 新建 `evidence/` 模块
- [ ] 实现 finding 规范化逻辑
- [ ] 实现 evidence 生成逻辑
- [ ] 实现 observation 聚合逻辑
- [ ] 实现冲突检测逻辑
- [ ] 实现 unknown / ambiguous 处理逻辑
- [ ] 为 evidence 与 observation 补充单元测试

### 推荐模块拆分

- `evidence/models.py`
- `evidence/normalize.py`
- `evidence/aggregate.py`
- `evidence/conflict.py`

### 完成标准

- 系统能够输出 `EvidenceItem[]`
- 系统能够输出 `NodeObservation[]`
- 当前 demo 中至少能展示 observation 结果

---

## 7. 子阶段 3：推理层替换准备

### 目标

让现有推理层从读取简单 `node_states`，逐步切换为读取 `NodeObservation[]`。

### 任务清单

- [ ] 为当前推理层增加 observation 输入适配器
- [ ] 设计新 BN 配置格式
- [ ] 对比并选择真实 BN 推理库
- [ ] 实现最小 BN 原型（至少覆盖 1~2 个风险维度）
- [ ] 输出 posterior 结构而不是只输出分数
- [ ] 为 legacy scoring 与新 BN 并行预留开关

### 建议策略

不要在第一步就完全删除旧逻辑，建议采用“双轨过渡”：

- 旧逻辑继续可运行
- 新逻辑先在部分维度试运行

### 完成标准

- observation 能稳定输入推理层
- 至少一条新的 BN 推理路径能跑通
- 当前系统可以并行比较旧逻辑和新逻辑

---

## 8. 子阶段 4：评估框架建立

### 目标

让系统具备最小可比较、可复现、可分析的能力。

### 任务清单

- [ ] 选定最小数据集范围（建议 NDA 子集）
- [ ] 从 `CUAD` / `ContractNLI` 中抽取可用样本，或自建小样本集
- [ ] 建立 extraction 评估脚本
- [ ] 建立 mapping 评估脚本
- [ ] 建立 risk report 对比脚本
- [ ] 增加 pure LLM baseline
- [ ] 增加 current system baseline
- [ ] 增加新中间层版本 baseline

### 最低需要记录的指标

- clause/finding 抽取准确率
- node observation 一致率
- overall risk 一致率
- top risk 命中情况

### 完成标准

- 能针对一个固定样本集跑出对比结果
- 能识别最主要误差来自哪一层

---

## 9. 子阶段 5：端到端跑通

### 目标

让新的中间层在现有系统形态下真正可运行。

### 任务清单

- [ ] 让 CLI 支持输出新的 evidence / observation 结构
- [ ] 让 demo 页面能看到新的中间层结果
- [ ] 让 API 支持返回 observation 信息
- [ ] 让报告层在保持兼容的前提下使用新的 risk trace 信息
- [ ] 增加至少一条端到端回归测试

### 完成标准

- 从合同文本输入到风险输出整链路可跑通
- 当前接口行为不至于完全破坏旧演示能力

## 11. 风险与应对

### 风险 1：语义设计反复变动

表现：

- node 不断新增删除
- state 定义频繁变化
- evidence 字段反复扩张

应对：

- 在子阶段 1 就冻结一版定义
- 第一版先覆盖最重要的节点，不求一步到位

### 风险 2：重构范围过大，导致系统失稳

表现：

- demo 跑不通
- API 大面积断裂
- 报告层无法消费新数据结构

应对：

- 采用双轨兼容策略
- 先加新层，不急于删旧层

### 风险 3：真实 BN 推理落地速度慢

表现：

- 技术选型耗时
- CPT 不易快速成型

应对：

- 先建立 observation 和 config 结构
- 再逐步替换推理内核
- 允许早期在部分维度先试点

### 风险 4：评估数据不足

表现：

- 难以判断系统是否真的变好

应对：

- 先建立小规模 gold set
- 以固定小样本做对比，先形成闭环

---

## 12. 推荐推进顺序

如果只考虑实际执行顺序，推荐按下列顺序推进：

1. 先冻结 node / state / evidence 定义
2. 再拆 evidence 模块
3. 再让推理层接 observation
4. 再做 BN 替换试点
5. 最后做 benchmark 与基线比较

这个顺序的核心逻辑是：

> 先稳住输入语义，再升级推理，不要一边改输入一边改所有输出层。

---

## 13. 阶段一验收标准

阶段一建议以以下标准作为验收条件：

### 结构层

- [ ] 已有正式 node schema
- [ ] 已有正式 evidence schema
- [ ] 已有正式 observation schema

### 实现层

- [ ] finding → evidence → observation 可运行
- [ ] observation → inference 可运行
- [ ] 至少一个风险维度完成新路径试跑

### 评估层

- [ ] 已建立最小 benchmark
- [ ] 已有至少两组 baseline 对比
- [ ] 已能定位主要误差来源

### 系统层

- [ ] CLI 可用
- [ ] demo 可展示新中间层结果
- [ ] API 不被重构完全破坏

---

## 14. 阶段一完成后的状态

当阶段一完成后，项目应从当前状态：

> LLM 抽取 + 直接映射 + 规则打分

升级为：

> LLM 抽取 + 标准化 Evidence Layer + Observation Layer + 可替换的 BN 推理骨架 + 基础评估闭环

这时项目就真正进入了“可信研究原型”阶段。

---

## 15. 结论

阶段一不是简单的代码重构，而是整个系统语义结构的重新立柱。

如果阶段一做得好，后续：

- 解释层增强
- 多合同类型扩展
- 产品化增强
- 学术实验对比

都会容易很多。

一句话总结本阶段：

> 阶段一的本质，是把项目从“演示系统”推进到“有研究价值的推理系统雏形”。
