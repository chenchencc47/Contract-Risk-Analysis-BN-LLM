# BN-Contract-Risk-Analysis 节点与证据层重构设计

## 1. 文档目标

本设计文档聚焦两个最关键的中间层：

- 节点体系（Node System）
- 证据层（Evidence Layer）

其目的不是简单扩充字段，而是为项目建立一个真正可以支撑概率推理、风险传播、解释追溯和后续评估的中间语义结构。

本设计文档承接已有文档：

- `01-推荐发展路线图.md`
- `02-目标架构设计.md`
- `03-分阶段待办清单.md`

并进一步回答以下核心问题：

1. 当前 node 为什么需要重构
2. 未来 node 应该如何分层
3. 当前 evidence mapping 为什么不够
4. 未来 evidence 应如何建模
5. LLM 抽取结果如何稳定地进入 BN 推理层

---

## 2. 当前问题诊断

### 2.1 当前 node 存在的问题

当前项目中的 node 同时混杂了多种语义：

- 有的 node 表示合同事实，例如是否存在某个条款
- 有的 node 表示法律语义，例如条款是否完整
- 有的 node 表示风险维度，例如财务暴露风险
- 有的 node 表示最终结论，例如 overall risk

这种混合方式能支撑 demo，但不利于未来扩展。主要问题包括：

1. 节点层级不清晰，事实、语义、风险、决策彼此交叉
2. 难以解释“哪个事实如何传导成风险”
3. 难以支持真正的 BN 依赖结构设计
4. 难以按合同类型扩展节点

### 2.2 当前 evidence mapping 存在的问题

当前 `build_evidence.py` 的核心逻辑是：

- 按 `finding_key`
- 或按 `clause_type`
- 或按 `risk_factor`
- 或按 `hypothesis`

将 findings 直接映射到 node state。

该方式的问题不是“错误”，而是“过薄”：

1. 没有独立 evidence 对象
2. 无法保留抽取的上下文语义
3. 无法区分“没抽到”和“确实缺失”
4. 无法表达歧义、冲突、不确定性
5. 无法支撑一条 evidence 影响多个 node 的情况

因此，未来不能继续采用“finding 直接进 node”的单薄结构，必须建设独立 Evidence Layer。

---

## 3. 重构目标

本次重构的目标不是做更多规则，而是做更清晰的语义结构。

### 核心目标

1. 把 node 变成有层级的风险推理图
2. 把 evidence 变成标准化、可追溯、可聚合的中间对象
3. 把 finding → evidence → observation → node 的路径固定下来
4. 把不确定性保留下来，而不是过早压平
5. 为真正 BN 推理和解释层打基础

### 设计原则

#### 原则一：事实与风险必须分层

合同中“有没有这个条款”和“这个条款是否带来高风险”不是一回事。

#### 原则二：证据必须可追溯

任何 node 状态都应能追溯到：

- 来自哪段合同文本
- 哪次 LLM 抽取
- 哪条映射规则

#### 原则三：保留不确定性

如果抽取不稳定，就不应硬性落为高风险或缺失。

#### 原则四：可扩展到多合同类型

当前先服务 NDA / 服务合同，但设计上不能绑死在某一类合同上。

---

## 4. 目标节点分层体系

建议将 node 重构为四层。

---

### 4.1 第一层：合同事实层（Contract Fact Layer）

这一层只回答一个问题：

> 合同文本中，事实层面出现了什么。

这一层 node 不直接表达风险，而表达客观合同事实。

建议节点示例：

- `termination_clause_existence`
- `liability_cap_existence`
- `governing_law_clause_existence`
- `dispute_resolution_clause_existence`
- `acceptance_clause_existence`
- `payment_clause_existence`
- `delivery_clause_existence`

这层节点适合的状态示例：

- `present`
- `missing`
- `unknown`
- `ambiguous`

说明：

- `present`：明确存在
- `missing`：在当前审查范围下可确认缺失
- `unknown`：没有足够信息判断
- `ambiguous`：文本可能提及，但语义不稳定

---

### 4.2 第二层：法律语义层（Legal Semantics Layer）

这一层回答的是：

> 即使条款存在，它在法律或商业语义上是否充分、平衡、清晰、对我方有利。

建议节点示例：

- `termination_clause_completeness`
- `termination_right_balance`
- `liability_cap_strength`
- `damages_exposure`
- `governing_law_clarity`
- `dispute_resolution_clarity`
- `jurisdiction_fairness`
- `acceptance_process_clarity`
- `payment_obligation_balance`
- `delivery_obligation_clarity`

这层状态不再只关心有无，而要表达“质量”和“倾向”。

建议状态示例：

- `acceptable`
- `unfavorable`
- `ambiguous`
- `unknown`
- 特殊场景下可保留更细状态，如 `severe` / `moderate`

这层是未来 BN 中最重要的一层，因为真正的风险大多不是由“有没有条款”决定，而是由“条款质量和倾向性”决定。

---

### 4.3 第三层：风险维度层（Risk Dimension Layer）

这一层负责聚合成稳定的业务风险维度。

建议保留的核心维度：

- `legal_enforceability_risk`
- `financial_exposure_risk`
- `performance_delivery_risk`
- `dispute_resolution_risk`
- `clause_balance_risk`

这层节点的状态建议统一成：

- `low`
- `medium`
- `high`

说明：

这一层是 BN 解释和报告层的主要桥梁。未来展示“风险维度”时，应主要来自这一层。

---

### 4.4 第四层：决策层（Decision Layer）

这一层面向最终系统输出，而非法律语义本身。

建议节点示例：

- `overall_contract_risk`
- `requires_manual_review`
- `signing_recommendation`

状态示例：

- `overall_contract_risk`: `low` / `medium` / `high`
- `requires_manual_review`: `yes` / `no`
- `signing_recommendation`: `sign` / `conditional_sign` / `do_not_sign`

说明：

这一层不应由上游 LLM 直接生成，而应由结构化推理层给出。

---

## 5. 节点元数据规范

未来每个 node 都应具备统一元数据结构，至少包含：

- `node_name`
- `node_layer`
- `description`
- `states`
- `parents`
- `contract_types`
- `business_meaning`
- `example_positive`
- `example_negative`

建议示例：

```json
{
  "node_name": "termination_clause_completeness",
  "node_layer": "legal_semantics",
  "description": "终止条款是否覆盖终止条件、通知周期及终止后责任",
  "states": ["acceptable", "unfavorable", "ambiguous", "unknown"],
  "parents": ["termination_clause_existence"],
  "contract_types": ["nda", "service_agreement"],
  "business_meaning": "影响合同退出机制和违约后的法律稳定性"
}
```

## 6. 状态语义重构

建议建立一个统一的状态语义框架，而不是每个 node 各自发散。

### 6.1 通用状态集

推荐优先采用以下六类通用状态语义：

- `present`
- `missing`
- `acceptable`
- `unfavorable`
- `ambiguous`
- `unknown`

不同层级允许裁剪，但不要跳出这组基础语义太远。

---

### 6.2 各状态定义

#### `present`
表示相关事实或条款明确存在。

#### `missing`
表示在当前文本和审查上下文下，能够较有把握地认定该条款或事实缺失。

#### `acceptable`
表示该语义项在当前合同中处于可接受状态，对我方风险相对可控。

#### `unfavorable`
表示该语义项对我方明显不利，或显著放大风险。

#### `ambiguous`
表示文本中存在相关内容，但表达模糊、适用范围不明、解释空间过大。

#### `unknown`
表示当前没有足够证据作出判断，不能将其简单当作缺失。

---

### 6.3 `missing`、`ambiguous`、`unknown` 的区分

这三个状态必须严格区分。

#### `missing`

适用场景：

- 经明确审查后，合同中没有该条款
- 文本覆盖范围足够，但仍未发现相关约定

#### `ambiguous`

适用场景：

- 文本疑似涉及该问题，但表达含混
- 条款存在但边界模糊
- LLM 与规则层提取结果存在语义摇摆

#### `unknown`

适用场景：

- 文本不完整
- chunk 覆盖不足
- 上游抽取不足以支持判断

未来不能再把这三类情况压平为同一个风险状态。

## 7. 证据层重构设计

### 7.1 Evidence Layer 的角色

Evidence Layer 的职责不是“再包装一遍 finding”，而是：

> 将 LLM 的原始抽取结果转成可用于结构化推理的标准证据对象。

它是 Extraction Layer 和 Bayesian Layer 之间的稳定桥梁。

---

### 7.2 EvidenceItem 设计

建议未来每条证据都抽象为 `EvidenceItem`。

推荐字段：

- `evidence_id`
- `contract_id`
- `source_chunk_id`
- `source_text`
- `source_span`
- `clause_type`
- `fact_type`
- `semantic_target`
- `raw_status`
- `normalized_status`
- `confidence`
- `ambiguity_flag`
- `supporting_reason`
- `extraction_model`
- `prompt_version`
- `created_at`

建议示例：

```json
{
  "evidence_id": "ev_termination_001",
  "contract_id": "nda-001",
  "source_chunk_id": "chunk_03",
  "source_text": "The agreement may be terminated by either party with 30 days notice.",
  "source_span": [128, 196],
  "clause_type": "termination",
  "fact_type": "termination_notice_period",
  "semantic_target": "termination_clause_completeness",
  "raw_status": "present",
  "normalized_status": "acceptable",
  "confidence": 0.88,
  "ambiguity_flag": false,
  "supporting_reason": "文本明确给出终止通知周期",
  "extraction_model": "kimi-k2.6",
  "prompt_version": "review_prompt_v2"
}
```

---

### 7.3 EvidenceSource 设计

建议区分 evidence 来源：

- `llm_extraction`
- `rule_derivation`
- `manual_annotation`
- `external_knowledge`

这样未来可支持：

- 自动抽取与人工标注并存
- 专家补录
- 外部知识规则辅助

---

### 7.4 NodeObservation 设计

EvidenceItem 不应直接等同于 node state。中间还应有 `NodeObservation`。

`NodeObservation` 表示：

> 经过 evidence 聚合与冲突处理后，对某个 node 的当前观察结论。

推荐字段：

- `node_name`
- `observed_state`
- `observation_confidence`
- `supporting_evidence_ids`
- `conflict_flag`
- `aggregation_strategy`

建议示例：

```json
{
  "node_name": "termination_clause_completeness",
  "observed_state": "acceptable",
  "observation_confidence": 0.82,
  "supporting_evidence_ids": ["ev_termination_001", "ev_termination_002"],
  "conflict_flag": false,
  "aggregation_strategy": "weighted_consensus"
}
```

## 8. Finding → Evidence → Observation 映射链

建议未来严格区分三个阶段。

### 阶段一：Finding 生成

由上游 LLM 生成原始 findings。

特点：

- 接近模型视角
- 粒度可粗可细
- 仍保留抽取噪声

### 阶段二：Evidence 规范化

将 findings 转为标准 EvidenceItem。

该阶段工作包括：

- 字段规范化
- clause_type 对齐
- raw_status 清洗
- 语义目标定位
- 置信度保留

### 阶段三：Observation 聚合

将多个 EvidenceItem 聚合为 NodeObservation。

该阶段工作包括：

- 同 node 多证据合并
- 冲突检测
- 歧义处理
- 最终 observation 输出

### 阶段四：BN 注入

将 NodeObservation 作为 observed evidence 注入推理层。

---

## 9. 多证据聚合策略

未来必须支持一个 node 被多条 evidence 命中的情况。

建议策略：

### 9.1 一致增强

如果多条证据指向同一状态，则提升 observation confidence。

### 9.2 冲突保留

如果多条证据给出不同状态，不应简单覆盖，而应：

- 标记 `conflict_flag`
- 记录所有 supporting evidence
- 根据规则决定是否落为 `ambiguous`

### 9.3 缺失优先级不能绝对化

当前用“严重度排序覆盖”的方式虽简单，但未来要更谨慎。

例如：

- 一条证据说 `present`
- 一条证据说 `unknown`

这不能简单让其中一个覆盖另一个，应根据来源与置信度决定。

### 9.4 推荐聚合方法

早期建议采用：

- 规则优先 + confidence 加权

后期可演进为：

- 证据加权共识
- 基于来源可信度的 observation 聚合

---

## 10. 与 BN 的衔接方式

未来 BN 不应直接读取 `finding_key_rules` 的输出，而应读取：

- `NodeObservation[]`

也就是说，BN 的输入不再是简单的 `node_states` 字典，而应是更强的 observation 对象。

建议 BN 输入至少支持：

- node name
- observed state
- observation confidence
- uncertainty marker

在第一阶段中，即使暂不做完整不确定性传播，也应该先在数据结构上预留这些能力。

---

## 11. 配置结构建议

建议未来将配置拆成三类，而不是全部混在一个大 JSON 中。

### 11.1 Node Schema 配置

负责：

- 节点定义
- 状态空间
- 层级信息
- 节点说明

### 11.2 Evidence Mapping 配置

负责：

- finding 到 evidence 的规范化规则
- evidence 到 observation 的映射规则
- clause_type / fact_type / semantic_target 的对齐规则

### 11.3 BN Structure 配置

负责：

- 节点关系图
- parent-child 结构
- CPT
- 阈值

这样做可以降低耦合，方便版本管理。

## 12. 与现有代码的重构关系

### 当前文件

- `review/ai_review.py`
- `pipeline/build_evidence.py`
- `bn/inference.py`

### 重构后建议

- `extraction/`：保留或迁移 `ai_review.py` 上游抽取能力
- `evidence/normalize.py`：finding → evidence
- `evidence/aggregate.py`：evidence → observation
- `bn/model.py`：BN 结构定义
- `bn/infer.py`：BN 推理执行

说明：

不要让 `build_evidence.py` 无限变厚，应该将其拆分成独立 evidence 模块。

---

## 13. 第一版建议落地范围

为了避免重构过大，第一版建议只做以下范围：

1. 完成四层 node 结构定义
2. 完成六类状态语义定义
3. 引入 `EvidenceItem` 和 `NodeObservation`
4. 将当前 mapping 流程改造成 finding → evidence → observation
5. 暂时保留现有风险维度输出接口，避免前端和报告层同时大改

也就是说：

> 第一版先重构中间层的数据语义，不急于一次性重写整个系统。

---

## 14. 结论

本项目未来是否能从 demo 升级为真正有价值的合同风险推理系统，核心取决于中间层是否做实。

节点层与证据层的重构，是整个系统演进中最重要的基础工程。

一句话总结本设计：

> 不再让 LLM 抽取结果直接撞上风险节点，而是通过标准化 Evidence Layer，将合同事实、语义判断与风险推理稳定地连接起来。
