# 报告稳定性优先与架构演进方案

> 日期：2026-05-08
> 适用范围：`LLM₁ 自由审查 → BN 一致性校验/反事实 → LLM₂ 最终报告` 当前主链路

---

## 一、要解决的核心问题

当前系统的主要矛盾，不是“提示词还不够长”，而是**同一份合同经过两层生成式判断后，最终报告存在漂移空间**：

- `LLM₁` 会以自由文本方式识别风险；
- `BN` 目前更多承担一致性校验、反事实模拟和概率量化；
- `LLM₂` 仍然拥有最终叙事主导权，会重新综合 LLM₁ + BN 输出最终报告。

这会带来三个直接问题：

1. **同一合同重复运行时，核心结论可能不稳定**；
2. **用户难以区分哪些结论是稳定事实，哪些只是 LLM 的法律表达**；
3. **用户信任会受损**——如果每次报告主结论都不同，系统即使“能写得漂亮”，也不够可信。

因此，这一阶段的首要目标不是继续堆 prompt，而是：

> **先稳结论，再优雅表达。**

---

## 二、总判断：优先做系统能力升级，而不是继续优先调 prompt

### 结论

未来方向应优先投入在**项目自身能力升级**，而不是把希望主要押在 prompt 微调上。

### 原因

#### 1. Prompt 只能改善表达，不能从根上消除“事实漂移”

你可以通过 prompt 约束 LLM 少发散、少改写，但只要最终事实裁决仍由 LLM 主导：
- 风险项是否保留；
- 风险等级如何定；
- 优先级如何排序；
- BN 数字如何解释；
- 最终签署底线如何落地；

这些关键结论就仍然会随着生成过程出现波动。

#### 2. 当前问题是架构问题，不是文案问题

现在的结构中，BN 没有成为“最终事实层”，只是“辅助验证层”。
这意味着系统最稳定、最可解释的一部分数据，没有真正接管最终口径。

#### 3. 用户信任建立在“重复运行结果稳定”上

法律/风控产品和普通内容生成产品不同。用户最在意的不是“这段话够不够像律师”，而是：
- 同一合同今天和明天的核心结论是否一致；
- 数字是否可追溯；
- 系统是否会把不确定性伪装成确定性。

---

## 三、推荐路线：稳定事实核 + LLM 解释层

推荐采用的不是“严格听 BN”或“让 LLM 自由发挥”这两个极端，而是第三条路：

> **LLM 负责理解合同与提供法律解释，BN/规则层负责稳定事实，最终报告由受约束的 LLM 渲染。**

### 1. LLM 继续负责的内容

- 从合同原文中识别候选风险；
- 提供法律解释、修改建议、谈判话术；
- 说明条款为什么危险；
- 把结构化结果写成可读、专业、像律师写的报告。

### 2. BN / 确定性层负责的内容

- 风险归一化与节点映射；
- 最终严重度和优先级规则；
- 反事实与概率数字；
- 跨维度联合风险；
- 人工复核触发条件；
- 最终对外的稳定事实口径。

### 3. 不应再由纯 LLM 单独决定的内容

- 最终接受哪些风险项；
- 每个风险项的最终严重等级；
- 最终优先级排序；
- 最终签署建议底线；
- BN 与法律判断冲突时的系统结论。

---

## 四、目标架构

目标是把系统演进为：

```text
合同原文
  ↓
LLM₁：自由审查（候选风险 + 法律解释）
  ↓
Canonicalization / Adjudication：风险归一化 + 稳定事实裁决
  ↓
BN：一致性校验 + 反事实 + 联合风险 + 冲突标记
  ↓
Report Dossier：结构化、确定性的报告事实清单
  ↓
LLM₂：受约束渲染器（只负责中文专业表达）
  ↓
最终报告
```

其中，**Report Dossier** 是关键：

- 它是系统对外的真实结论；
- 它必须是结构化、确定性、可复验的；
- LLM₂ 只能“翻译”它，不能“改判”它。

---

## 五、分阶段实施路线

## Phase A：先稳结论（当前最高优先级）

### 目标
先把“同一合同多次跑，核心结论漂移”压下来。

### 具体动作
1. 把 `LLM₂` 从“最终判断者”降级为“受约束渲染器”；
2. 将最终事实源从 narrative report 改为结构化 dossier；
3. 降低 `LLM₂` 温度到接近 0，并固定输出框架；
4. 明确禁止 `LLM₂`：新增/删除风险项、改数字、改严重度、改优先级、改签署底线；
5. 对 BN/LLM 冲突项统一输出“建议人工复核”，而不是让 LLM₂ 自由选边。

### 为什么先做这个
- 改动杠杆最大；
- 对现有代码侵入相对可控；
- 能直接提升用户对系统结果的信任；
- 不会立刻牺牲律师式表达能力。

---

## Phase B：稳定 LLM₁ → BN 的交接

### 目标
减少上游自由文本波动向 BN 层传播。

### 具体动作
1. 将 `RiskSegment.clause_type` 从完全自由文本收紧为“自由文本 + canonical_type”；
2. 在 `LLM₁` 输出后新增确定性的 canonicalization 层；
3. 归并同义标签、不同写法和重复风险项；
4. 为每个风险项绑定稳定证据摘录和统一 `issue_id`；
5. 让 `BnMappingService` 优先消费 canonical 字段，而不是依赖自由文本 heuristic。

### 价值
这一步会显著降低 `bn_mapping.py` 因文本写法变化导致的节点映射漂移。

---

## Phase C：新增裁决层（Adjudication Layer）

### 目标
让“最终风险事实”成为系统属性，而不是模型临场发挥的结果。

### 放置位置
位于：
- `free_review_contract_text()` 之后；
- `generate_combined_report()` 之前。

### 职责
- 合并重复 finding；
- 分配统一 issue id；
- 用规则决定最终 severity / priority；
- 结合 BN 输出生成稳定 dossier；
- 对证据不足或系统冲突项打上人工复核标记。

---

## Phase D：长期产品形态

### 目标
把产品做成“可信法律 Copilot”，而不是“会写漂亮报告的生成器”。

### 最终建议输出两层结果
1. **Deterministic Fact Sheet**：风险表、证据、严重度、优先级、BN 数字、复核标记；
2. **Lawyer Memo**：基于 fact sheet 自动生成的自然语言报告。

这样即使叙述略有变化，用户也知道稳定核心没有变。

---

## 六、关键文件落点

### 第一阶段最值得优先改的文件

- `backend/main.py`
  - 当前运行入口，需要承接稳定事实层的输出与返回。
- `src/contract_risk_analysis/review/report_writer.py`
  - 当前 LLM₂ 仍有最终判断权，是 Phase A 的最高杠杆点。
- `src/contract_risk_analysis/review/ai_review.py`
  - LLM₁ 输出结构、canonicalization 接入点、prompt 约束增强。
- `src/contract_risk_analysis/bn/bn_mapping.py`
  - 需逐步从 heuristic 映射过渡到 canonical 映射。
- `src/contract_risk_analysis/bn/consistency_validator.py`
  - BN 输出进入 dossier 的关键入口。
- `src/contract_risk_analysis/domain/free_review_schema.py`
  - 需要承载 canonical issue、稳定字段和 dossier 结构。
- `src/contract_risk_analysis/review/verification.py`
  - 可复用作冲突项/低质量项的定点复核机制。

## 七、分层一致性目标（2026-05-08 外部咨询确认）

> 外部咨询给出的结论与本方案总体一致：**不需要追求整份报告逐字逐句完全一样，但必须锁定事实层和核心判断层的一致性；允许论证路径和措辞存在有限自然差异。**

本方案在此基础上把一致性拆为三层，明确"哪些必须锁死、哪些可以自然变化"：

| 层级 | 内容 | 一致性要求 | 说明 |
|------|------|-----------|------|
| **事实层** | 条款是否存在、条款位置、关键数字提取、BN/规则生成的结构化字段 | 接近绝对一致 | 不准漂。这是系统的"眼睛"。 |
| **核心判断层** | 风险是否入选、最终 severity、最终 priority_rank、signing advice 底线、人工复核标记 | 稳定一致 | 不准漂。这是系统的"判断力"。 |
| **表达层** | 法律论证展开顺序、解释角度、行文措辞、谈判话术表述 | 允许有限自然变化 | 可变化，但不能引入新事实或改判核心结论。 |

**特别强调**：单次报告内部一致性（同一份报告前后不打仗）与跨轮稳定性（同一合同多次运行结论不漂）同等重要。

---

## 八、验证标准

### 1. 重复运行稳定性验证（核心）
对同一份合同固定输入，连续运行 10-20 次，比较以下字段是否完全一致：
- 风险项集合
- `issue_id`
- `severity`
- `priority_rank`
- BN annotations
- counterfactual 数字
- signing advice 底线

**目标**：这些字段必须完全一致；允许 narrative 措辞有少量自然变化。

### 2. 单次报告内部一致性验证
- 检查同一份最终报告内：
  - 执行摘要、风险总览、逐项分析中的 severity 是否一致；
  - 谈判优先级、签署底线、整改建议是否互相冲突；
  - narrative 是否前文认定高风险、后文降格处理，或前文要求保留、后文又作为可交易项放弃。
- 目标：消除"单次报告自己打架"的问题。

### 3. Narrative Guardrail 验证
检查最终报告是否引入 dossier 中不存在的新风险、新数字或新结论。

**目标**：0 次越权扩写。

### 4. 映射稳定性验证
同一风险项使用不同同义 `clause_type` 文案，验证是否仍映射到同一 canonical issue / BN node。

**目标**：同义表述不改变 BN 结论。

### 5. 冲突升级验证
构造 LLM 与 BN 冲突、证据不足、条款缺失判断不稳的样本。

**目标**：系统输出“人工复核”，而不是随机给不同结论。

### 6. 端到端回归验证
使用现有买卖合同、销售合同等样本跑完整链路。

**目标**：
- 核心风险仍被识别；
- 报告显著更稳定；
- 说明文字仍保持法律专业性。

---

## 九、最终建议

### 明确建议 1
**先做系统能力升级，不要把主要精力继续放在 prompt 微调上。**

### 明确建议 2
**不要让 BN 完全接管整份法律报告，也不要继续让 LLM₂ 自由综合。**

### 明确建议 3
**第一阶段优先“先稳结论”**：
- 先保证风险项、优先级、BN 数字、签署底线稳定一致；
- 再优化文风、节奏感和律师表达风格。

### 一句话总结

> 未来最正确的方向，是把系统做成“稳定事实核 + 受约束解释层”：
> **BN/规则/裁决层掌握稳定事实主权，LLM 保留合同理解和律师式表达能力。**

这条路线最能同时解决：
- 报告漂移；
- 用户信任；
- 法律质量；
- 项目未来可扩展性。
