# 合同类型智能路由与公司红线方案

> 版本：v1.0 | 日期：2026-04-30 | 状态：待实施

---

## 一、问题

### 1.1 当前痛点

BN 有 57 个 contract_fact 节点，LLM₁ 每次审查时看到全部 57 个。但实际上一份购销合同只需关注 ~20 个节点，剩下的 37 个（源代码托管、竞业限制、热值计价等）全是噪音。

噪音的后果：LLM₁ 注意力被稀释，有时会静默跳过真正重要的节点（如 report-11 漏掉了质保期）。

### 1.2 设计目标

1. 每份合同审查时，LLM₁ 只看到与合同类型相关的检查清单——减少噪音，提高覆盖率
2. 企业专属规则（红线）可以配置并自动注入审查流程
3. 不引入多 agent 架构，保持当前管线简单

---

## 二、架构概览

```
合同文本 → 合同类型检测 → 过滤 BN 检查清单 → 注入公司红线 → LLM₁ 审查 → BN 校验 → LLM₂ 报告
              │                    │                   │
              ▼                    ▼                   ▼
        关键词匹配          只展示相关的          从 YAML 读取
        或 LLM 判断         ~20/57 节点           企业专属规则
```

新增两个文件，改动两个文件：

| 文件 | 类型 | 说明 |
|------|:--:|------|
| `config/contract_type_routing.yaml` | 新建 | 合同类型 → 节点子集映射 |
| `config/company_redlines.yaml` | 新建 | 按合同类型分组的公司红线规则 |
| `review/ai_review.py` | 改动 | `_build_bn_checklist()` 支持按合同类型过滤 |
| `backend/main.py` | 改动 | 管线中加入合同类型检测 + 红线注入 |

## 三、合同类型检测

### 3.1 检测方式

两级检测，无需 LLM 额外调用：

1. **关键词匹配（零成本）**：扫描合同文本前 500 字，匹配关键词表
2. **LLM₁ 自由审查时附带判断（已有）**：LLM₁ 已经在做 free review，顺便判断合同类型

```python
# 关键词映射表
CONTRACT_TYPE_KEYWORDS = {
    "销售合同": ["购销", "买卖", "出售", "供货", "瓷砖", "建材"],
    "采购合同": ["采购", "购买", "供应商", "原料", "设备采购"],
    "煤炭合同": ["煤炭", "热值", "发热量", "收到基", "空干基", "kcal"],
    "NDA": ["保密", "机密信息", "非披露", "NDA", "保密协议"],
    "软件许可": ["软件", "许可", "源代码", "授权", "SaaS", "license"],
}
```

### 3.3 节点去重 + 合并

如果合同同时匹配多个类型（如"瓷砖原料采购"同时匹配"销售合同"和"采购合同"），取两者的节点并集。不冲突。

## 四、公司红线配置

### 4.1 文件格式

```yaml
# config/company_redlines.yaml
煤炭购销:
  - id: coal_calorific_basis
    规则: 煤炭计价必须以到厂收到基低位发热量为准，不接受空干基或矿发量
    来源: 行业标准 GB/T 213
    级别: 致命
    
  - id: coal_trial_burn
    规则: 必须约定试烧验收条款：试烧不合格的，买方有权退货，卖方承担全部运输费用
    来源: 行业惯例
    级别: 致命
    
  - id: coal_measurement
    规则: 计量方式以买方到厂计量衡为准，不接受卖方单方计量结果
    来源: 国家标准
    级别: 高

瓷砖销售:
  - id: tile_liability_cap
    规则: 责任上限不超过合同总价的 100%，必须排除间接损失
    来源: 民法典第 584 条 + 公司风险政策
    级别: 企业红线
    
  - id: tile_acceptance_period
    规则: 外观验收至少 7 个工作日，内在质量异议期至少 30 天
    来源: 行业惯例
    级别: 高

通用:
  - id: no_unlimited_liability
    规则: 任何合同不得包含无限赔偿责任条款
    来源: 公司风险政策
    级别: 企业红线
```

### 4.2 注入方式

管线在调用 LLM₁ 前，根据检测到的合同类型，读取对应的红线规则，追加到 prompt 中：

```
## 公司红线（不可妥协的底线原则）

以下规则来自公司风险政策，审查时必须逐条对照：
- [企业红线] 任何合同不得包含无限赔偿责任条款
- [企业红线] 责任上限不超过合同总价的 100%，必须排除间接损失
- [高] 外观验收至少 7 个工作日，内在质量异议期至少 30 天
```

LLM₁ 不需要"执行"这些红线——它的职责是**检查合同是否违反红线，并在报告中标注**。

## 五、管线集成

### 5.1 改动点

`_run_v2_pipeline` 中增加两步：

```python
# Step 0: 合同类型检测（在 LLM₁ 调用之前）

# Step 0.5: 加载公司红线

# Step 1: LLM₁ 自由审查（传入过滤后的检查清单 + 红线）

    contract_type=contract_type,       # 新增参数
    redlines=redlines,                  # 新增参数

```

## 六、与原方案的对比

| | 原方案（universal/conditional） | 本方案（合同类型路由） |
|------|------|------|
| 节点分类方式 | 预设通用/条件标签 | 按合同类型关联节点子集 |
| 分类由谁决定 | 我人工标注 | routing 配置文件 + 关键词自动匹配 |
| 每份合同审查时 LLM₁ 看到的节点数 | 57 | ~20 |
| 新增合同类型时 | 需要判断每个节点属于哪类 | 在 routing 配置中新增一组 nodes 列表 |
| 公司红线 | 无 | YAML 配置，按合同类型分组 |

## 八、未来扩展

- **红线规则超过 50 条**：升级为 RAG（向量检索），但仍保留 YAML 作为规则源
- **合同类型超过 10 个**：routing 配置仍可维护，不需要 agent 架构
- **需要跨合同类型对比**：全部节点并集展示，不限于单个类型的子集
