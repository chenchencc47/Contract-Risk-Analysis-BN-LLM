## 三、各维度详细分析

### 1. 核心算法/方法创新性 — 8.5分

**亮点：**

- **BN作为一致性校验器(而非评分器)的定位**是核心创新。市面上的 Kira/Luminance/Evisort 都是"AI提取→打分"的线性管道，本系统将 LLM 的自由审查与 BN 的结构化推理**正交化**——LLM负责"能发现什么"，BN负责"发现得对不对/全不全"——这是学术界和工业界都在探索的方向。
- **Noisy-OR/Noisy-MAX CPT 生成** (P4.4) 将 overall_contract_risk 的 CPT 从 3^5=243 行压缩到 15 个参数，这是一个扎实的概率图模型工程优化。
- **反事实敏感性分析**：逐个翻转证据节点状态，计算整体和维度级的高风险概率变化量，这在合同审查领域几乎没有类似产品提供。
- **乘数效应风险检测**：预定义了4组跨维度风险组合规则（如"高财务暴露 + 高争议处置 → 追偿无门"），这是BN因果结构的自然延伸。
- **双层反事实数据**：整体级(展示方向) + 维度级(有区分度的量化证据)的分层报告策略，解决了 Noisy-OR 聚合后整体概率压缩的呈现问题。

**扣分项：**

- Noisy-OR 假设维度间**独立因果影响**，但乘数效应检测本身已证明某些组合的"危害远超简单加总"，而BN本身并没有建模这些交互效应。这是概率图模型的结构性限制，可以考虑加入 interaction terms 或使用更复杂的影响模型。
- 证据节点→BN状态映射的3层匹配(显式suggested_bn_nodes → finding_key反向查询 → heuristic)有大量硬编码映射表，如 CLAUSE_TYPE_HINTS 含140+条中英文关键词映射，维护成本高且覆盖不完整。

---

### 2. 工程实现质量 — 7.5分

**亮点：**

- **清晰的分层架构**：`bn/` `evidence/` `pipeline/` `review/` `evaluation/` `domain/` 各司其职，依赖关系单向可控。
- **良好的抽象**：`RiskEvidence` → `RiskReport` → `PolishedReport` 的数据流链路清晰，每层有明确的转换逻辑。
- **配置驱动设计**：BN结构完全通过 `bayesian_network_v2.json` 定义，新增节点无需改代码。
- **优雅的适配器模式**：`free_review_to_legacy_result()` 实现了 v2→v1 的向后兼容转换，`register_calibration_source()` 实现了数据源的可插拔注册。
- **CPT缺失值的多层fallback**：`unknown` 状态自动推断 → 均匀分布兜底，不允许未定义的 CPT 组合通过。

**扣分项：**

- `inference.py` 文件 454 行，包含了 legacy/rich/pgmpy 三套推理逻辑的混杂，函数名 `_legacy_assess_risk` / `_assess_rich_risk` / `_assess_risk_pgmpy` 暴露了多次重构的痕迹。应该按策略模式分离。
- `consistency_validator.py` 837 行，包含 `BnMappingService` + `BnValidator` + 全局常量 + orchestrator，粒度太粗。
- 全局常量和字面量散落多处：`DIMENSION_LABELS` 在 `inference.py` / `consistency_validator.py` / `report_writer.py` / `server.py` 中各自定义了一份。
- Python package 名 `contract_risk_analysis` 与项目名 `bn-contract-risk-analysis` 不一致。

---

### 3. 数据驱动 & 可验证性 — 8.0分

**亮点：**

- **每个概率数字有可追溯来源**的原则在 CLAUDE.md 中形成了正式文档化约束，且 CPT 文件中确实标注了 `cpt_source` 字段（`contractnli_empirical` / `cuad_empirical` / `expert_estimated`）。
- **ContractNLI 校准管道**（`cpt_calibrator.py`）：从 6,819 条标注数据中计算先验概率和转移概率，替换 expert_estimated 值，有完整的 `calibration_report` 输出。
- **置信度校准参数**（`compute_confidence_calibration_params`）：利用 ContractNLI 标签分布推断法律问题的"难度"，用于检测 LLM 置信度异常。
- **行业数据集对齐**：CUAD (510份合同, 41类条款标注) 和 ContractNLI 是合同NLP领域的标准基准数据集，这为对标学术研究提供了可能。
- **外部评价记录**：有 DeepSeek 等其他模型的交叉验证评价。

**扣分项：**

- CUAD 数据仅用于扩展节点清单（node_schema 中的 30 个 CUAD 节点），**但 CUAD 的标注数据并未被充分利用于 CPT 概率校准**——目前真正做了 empirical calibration 的只有 ContractNLI 的 confidentiality 节点。
- `compute_confidence_calibration_params()` 的 `baseline_accuracy_estimate = 0.75` 是一个标注为 "Placeholder" 的占位值，说明置信度校准并没有经过真正的 LLM benchmark 验证。
- 缺乏与 golden-standard 人工审查的对比基准。

---

### 4. 实用性与可落地性 — 6.5分

**亮点：**

- 已产出完整的法务风格审查报告（见 `contract-review-v2-p0.md`），7章结构(执行摘要/风险总览/逐条款分析/反事实分析/签署建议/整改计划/方法论)，语言专业度接近真实法律意见书。
- 双重校验机制(LLM审查 + BN一致性验证)增强了可信度，在报告中有明确的"分歧处理"和"方法论说明"章节。
- 有 CLI、FastAPI 后端、React 前端、Streamlit Demo 四种交互方式，入口丰富。
- 针对具体合同类型(销售合同、瓷砖采购合同)有定制化审查示例。

**扣分项：**

- **支持的合同类型有限**：BN 的网络结构主要针对销售/采购合同，CUAD 节点覆盖的 30 个类型(如 IP ownership, non-compete, revenue sharing)虽然已加入清单，但相应的 CPT 并未校准，实际推理效果未知。
- **无法处理大型合同**：LLM₁ 的 `max_tokens=8192` 限制意味着合同文本不能超过约 6,000 中文字。真实商业合同动辄数万字。
- **无增量审查能力**：每次审查都是完整流程，无法追踪同一合同的多版本演变。
- **无用户认证/权限系统**：当前只适合本地 demo，不适合企业部署。
- **依赖双 LLM API 调用**：每次完整审查需要使用 Kimi(LLM₁) + DeepSeek(LLM₂)，成本较高且增加了延迟。生产环境中这是一个显著的成本因素。

---

### 5. 前端UX/交互 — 6.0分

**亮点：**

- React + TypeScript 现代技术栈，组件结构清晰（`Header` / `ContractInput` / `RiskReport` / `RiskGauge` / `AttributionChain` / `ActionPlan`）。
- 加载状态有渐进式动画提示（4个步骤：LLM审查→BN校验→反事实模拟→生成报告），符合用户等待心理。
- 暖色调法律文书风格（`#FAF8F5` / `#8B6F5C`），视觉调性与法务场景匹配。

**扣分项：**

- 前端组件数量有限(9个)，报告展示更多是数据dump而非交互式探索——无法点击某个风险项深入展开、无法切换查看维度、无法比较反事实场景。
- 无反事实场景的交互式 slider/对比视图，而这恰恰是核心差异点——"改这个条款能让高风险降X%" 是极有价值的洞察，但在前端没有得到充分的可视化呈现。
- 无合同文本高亮对照（highlight the exact clause in the original text），只能看到摘录文本。

---

### 6. 文档与可维护性 — 8.5分

**亮点：**

- **CLAUDE.md 极为出色**：包含4条核心原则(每个概率数字可追溯/提方案前先查证据/区分真问题和呈现问题/数字溯源链)、项目约定(文件路径/测试命令/报告版本管理)、详细的犯错记录及其根因分析。这是"living documentation"的典范。
- **WORKLIST/PROGRESS 分明**：工作清单和进度记录分文件维护，有明确的优先级标记(P0/P1/P2)。
- **策略文档完备**：有 ROADMAP、TARGET-ARCHITECTURE、分阶段待办清单、节点与证据层重构设计等完整技术文档。
- **报告命名规范**：`contract-review-{合同类型}-{序号}.md`。
- **prompt 中有严谨的约束**：LLM₂ prompt 中明确"不得修改或虚构任何风险分数"、"不得编造法律条文"等约束。

**扣分项：**

- README 过于简陋（仅1行英文），与项目的实际复杂度严重不匹配。
- 缺少 API 文档（虽然 FastAPI 有 /docs，但未定制）。
- 部分中文/英文混用的代码风格不够一致。

### 7. 测试质量 — 7.0分

**亮点：**

- 19 个测试文件，覆盖了 BN/CLI/evidence/pipeline/review/web 等核心模块。
- 针对 BN 推理有专门的 `test_noisy_or.py` 和 `test_inference.py`。
- 有 `test_build_evidence_rollout.py` 这种端到端的流程测试。

**扣分项：**

- 已知有 4 个 Streamlit demo 测试预存失败(CLAUDE.md 中注明)。
- 测试没有覆盖实际的 LLM API 调用（由于依赖外部API，这是合理的，但应标注为集成测试范畴）。
- 缺少针对 BN 一致性验证器的单元测试。

## 五、核心竞争力总结

本项目的最大优势在三个维度上形成了差异化护城河：

1. **BN因果推理**（竞品几乎空白）—— 不是简单地"这个条款有风险"，而是"这个条款异常→驱动X维度风险升高→与Y维度形成乘数效应"的因果链。
2. **反事实模拟**（竞品完全没有）—— "如果修改付款条款，财务暴露风险预计从62%降至22%"这种量化洞察，当前市面上没有任何合同审查产品提供。
3. **可解释性归因链**（竞品普遍薄弱）—— 从证据摘录→节点状态→维度概率→整体风险的全链路透传，有明确的数字溯源。

最大短板是**规模化能力**(合同长度限制/双API延迟)和**企业就绪度**(无认证/权限/审计/私有化部署方案)，这些是将学术原型转化为商业产品需要攻克的关键障碍。

**综合评价：作为学术/研究型项目，达到了较高水准（7.6/10）。若以商业化产品标准衡量，约 5.5-6/10，核心差距在于规模化工程和企业级能力。**
