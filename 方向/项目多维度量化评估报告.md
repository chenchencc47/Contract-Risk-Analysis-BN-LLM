## 二、视角一：个人项目评分 (87/100)

评价标准：技术架构、代码质量、创新能力、工程完整度、数据严谨性、文档质量。

### 2.1 技术架构与设计 — 23/25

| 子项 | 得分 | 评价 |
|------|:---:|------|
| 三层BN结构（fact→semantics→dimensions→overall） | 5/5 | 学术级设计，因果层级清晰 |
| LLM+BN融合管线（LLM做语义、BN做概率一致性） | 5/5 | 职责分工优雅，不是简单的"LLM打分+汇总" |
| CPT数据溯源体系（三类合法来源标注） | 4/5 | 理念极好，但约40%节点仍为expert_estimated |
| Noisy-Max聚合模型 | 5/5 | 避免了全CPT枚举的组合爆炸问题（3^5=243行→15参数） |
| 合同类型分层路由+公司红线配置化 | 4/5 | 设计思路好，但路由是纯关键词匹配（无NLP分类器） |

**亮点**: pgmpy变量消元实现反事实模拟、联合概率乘数效应分析、双视角审查管线、后处理规则校验——这些在个人项目中很罕见。

**扣分**: 合同类型识别纯关键词匹配；`backend/main.py` 660行单文件略臃肿。

### 2.2 代码质量与工程规范 — 15/20

| 子项 | 得分 | 评价 |
|------|:---:|------|
| 模块化结构 | 4/5 | `bn/`、`review/`、`evidence/`、`evaluation/` 分层清晰 |
| 类型安全 | 4/5 | Python端全面使用类型标注+dataclass；前端TypeScript |
| 测试覆盖 | 2/5 | 1435行测试，覆盖核心模块，但缺集成测试和端到端测试 |
| 错误处理 | 2/3 | 存在多处 `except Exception: pass` 静默吞错 |
| 代码重复 | 3/2 | `bn_mapping.py` 的 `CLAUSE_TYPE_HINTS` 是170行的硬编码映射表，维护负担重 |

**扣分**: 关键路径缺集成测试；巨型硬编码字典；多处静默异常吞没。

### 2.3 创新性与差异化 — 19/20

| 子项 | 得分 | 评价 |
|------|:---:|------|
| BN作为一致性校验器而非打分器 | 5/5 | 核心理念创新，区别于所有纯LLM方案 |
| 反事实模拟（"修掉X条款，风险降Y%"） | 5/5 | 在合同审查领域几乎未见同类功能 |

## 三、视角二：报告输出质量评分 (92/100)

基于最新输出 `contract-review-买卖合同-2.md`（DeepSeek评估：95/100）的独立评分。

> 注意：此评分仅评价报告质量，不评价产生报告的代码质量。

### 3.1 风险识别全面性 — 23/25

- 15项风险覆盖了付款结构、责任上限、交付验收、质保期、保密、知识产权、不可抗力等关键领域
- 2项遗漏：法律文书送达地址、合同生效条件

### 3.2 风险分析深度 — 24/25

- "无责任上限是双刃剑"的洞察——穿透表面有利条款识别隐藏风险
- 预付款80%与质保金5%的结构性失衡分析
- BN乘数效应场景化解读（"投入了钱、拿不到好产品、还难以追回损失"）
- 民法典第584/585/622条的具体引用和推理

### 3.3 解决方案实操性 — 24/25

- 第五章"筹码防御与谈判策略"是最大亮点：
  - 精准识别两个核心筹码（无责任上限 + 高额违约金）
  - 预判对手律师的具体攻击话术（逐字逐句）
  - 三层防御策略（防守话术→交换筹码→底线划定）
  - 每个交换方案有具体数字、条款号、可执行措辞
- 签署建议给出了清晰的"签约底线"和"可接受最终方案"

### 3.4 方法论严谨性 — 13/15

- BN反事实8项，每项有维度级数据+交叉校验判断
- 正确区分了"维度级数据（主要）"和"整体概率（辅助参考）"
- 坦诚说明了BN局限性（单维度假设、统计先验局限）
- 扣分：4.7/4.8两项反事实（管辖、争议解决）delta仅12-13%，对决策帮助有限，属于"为用完BN数据而写"

### 3.5 报告可读性与结构 — 8/10

- 八章结构完整，Markdown排版清晰
- 表格运用得当（风险全景表、五维评分卡、优化建议整合表）
- 扣分：全文约26000字，执行摘要可更精炼；第4章8项反事实之间逻辑关联不够紧密

5. `pgmpy_adapter.py` 的 `favorable_states` — 加最优状态推断
6. 可能还需改 `ai_review.py` 的 prompt

## 七、从63到80：开源化优先级路线图

### 第一周：冲击 +10分（63→73）

**P0-1: Docker Compose 一键启动**（4分）

```yaml
# docker-compose.yml — 开源项目转化率最高的单一文件
services:
  backend:
    build: .
    ports: ["9527:9527"]
    environment: ...
  frontend:
    build: ./frontend
    ports: ["5173:5173"]
  mysql:
    image: mysql:8.0
```

效果：`git clone + docker-compose up + 浏览器打开localhost:5173` — 5分钟看到效果。

**P0-2: `.env.example` + SQLite demo 模式**（3分）

- 提供 `.env.example` 模板
- Demo模式：预设合同文本 + 预计算BN结果 → 不依赖MySQL和API Key即可体验核心推理

**P0-3: README 英文版**（3分）

不需要全文翻译，只需：
- 一句话核心思路
- 架构图
- Quick Start（docker-compose up）
- 示例报告截图

### 第一个月：冲击 +7分（73→80）

**P1-1: 拆分 `backend/main.py`**

```
backend/
  main.py          # FastAPI app 创建 + CORS
  routers/
    review.py      # /api/review, /api/v2/review, /api/v2/review/dual
    export.py      # /api/export/pdf, /api/export/md
    history.py     # /api/reports, /api/reports/{id}, /api/reports/diff
    sandbox.py     # /api/bn/nodes, /api/bn/simulate
    redlines.py    # /api/redlines CRUD
    feedback.py    # /api/feedback
    upload.py      # /api/upload
    health.py      # /api/health
```
